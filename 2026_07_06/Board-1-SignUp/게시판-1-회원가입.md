# JPA로 회원가입 만들기 (계층구조 + 예외 공통화)

> 스프링 부트로 **첫 CRUD**를 시작해요. 회원가입 한 기능을 만들면서, 스프링 웹앱의 **계층 구조**(컨트롤러 → 서비스 → 리포지토리)와 **JPA로 DB에 저장**하는 흐름, 그리고 **예외를 한 곳에서 처리**하는 법까지 한 번에 익혀요.
> **아래 Step을 순서대로 따라가면, 화면에서 가입 버튼을 눌러 DB에 회원이 저장되는 회원가입이 완성됩니다.**
>
> 각 Step의 **힌트는 접혀 있어요.** "이 계층은 무슨 일을 하지?"를 먼저 떠올리고, 막히면 "힌트 보기"를 펼치세요.
>
> 이번 과제는 **스프링 시큐리티를 아직 안 배운 단계**라, 비밀번호를 **평문 그대로** 저장해요. (실무에서는 절대 이렇게 하지 않아요. 암호화는 뒤에서 배워요.)

---

## 0. 먼저 알아둘 점

- 이 과제는 **스프링 부트 프로젝트가 이미 세팅됐다는 전제**예요. 아래는 제공돼요:
  - `build.gradle` 의존성: `spring-boot-starter-data-jpa`, `spring-boot-starter-web(mvc)`, `spring-boot-starter-thymeleaf`, `mysql-connector-j`, `lombok`
  - `application.yaml`의 **DB 접속 정보**(MySQL `java_basic`), JPA 설정
  - **프론트 화면**(`sign-up.html`)과 **자바스크립트**(`signUp.js`) — 여러분은 화면을 건드리지 않아요
  - 회원가입 폼을 보여주는 **뷰 컨트롤러**(`GET /members/join`)
- 실행하려면 **MySQL 서버**가 켜져 있고 `java_basic` DB가 있어야 해요. `member` 테이블은 Step 1에서 JPA가 만들어 주게 설정해요.
- **여러분이 작성할 것**은 화면 뒤의 **백엔드 계층**이에요: 엔티티 · 리포지토리 · DTO · 매퍼 · 서비스 · 컨트롤러 · 예외 처리.

> 프론트(`signUp.js`)가 이미 정해둔 **약속(계약)** 이 있어요. 여러분의 백엔드는 이 약속에 맞춰야 해요:
> - 요청: `POST /api/members/join` 으로 **JSON** `{ "userId": "...", "password": "...", "userName": "..." }` 을 보냄
> - 성공 응답: `{ "url": "/members/login" }` → JS가 그 주소로 이동시킴
> - 실패 응답(중복): 상태코드 **409** + `{ "message": "..." }` → JS가 그 message를 alert로 띄움

## 1. 무엇을 만드나요?

회원가입 폼에 아이디/비밀번호/이름을 넣고 **가입**을 누르면 → **DB `member` 테이블에 저장**되고 → **로그인 페이지로 이동**해요. 이미 있는 아이디면 **경고창으로 안내**하고 이동하지 않아요.

| 주소(메서드) | 하는 일 | 계층 |
|------|------|------|
| `GET /members/join` | 회원가입 폼 보여주기 | 뷰 컨트롤러 (**제공**) |
| `POST /api/members/join` | 가입 처리(중복검사 → 저장) | **여러분이 만들 API** |

**완성되면 이렇게 동작해요**
```
[가입 성공]
  아이디: newbie / 비번: 1234 / 이름: 새싹  → [가입]
  → member 테이블에 새 행 저장 → 로그인 페이지로 이동

[가입 실패 - 중복]
  아이디: hong (이미 있음) → [가입]
  → 경고창: "이미 존재하는 아이디입니다."  (이동 안 함)
```

## 3. 핵심 개념

### (1) 계층 구조 — 요청이 흐르는 길

```
[브라우저] --JSON--> Controller ---> Service ---> Repository ---> [DB]
                     (웹 담당)     (로직 담당)   (저장 담당)
```

| 계층 | 책임 | 이번 과제에서 |
|------|------|------|
| Controller | HTTP 요청/응답 처리 (JSON 받고 주기) | 가입 요청을 받아 서비스에 넘김 |
| Service | 비즈니스 로직 | **아이디 중복 검사** 후 저장 지시 |
| Repository | DB 접근 | INSERT / 존재여부 조회 |

> 왜 나누나요? 각 계층이 **한 가지 일**만 하면, 바뀔 때 그 계층만 고치면 되고 테스트도 쉬워져요.

### (2) JpaRepository — 인터페이스만 있는데 어떻게 동작하지?

```java
public interface MemberRepository extends JpaRepository<Member, Long> { }
```
- 구현 클래스를 **우리가 안 만들어요.** 스프링 데이터 JPA가 뜰 때 **구현체를 자동 생성**해 빈으로 등록해요.
- `JpaRepository<Member, Long>` = "`Member` 엔티티를, 기본키 타입 `Long`으로 다루는 리포지토리".
- 상속만 해도 `save` / `findById` / `count` / `existsById` … 가 **공짜로** 딸려와요.
- 조건 조회는 **메서드 이름 규칙**으로: `existsByUserId(String)` → `SELECT ... WHERE user_id = ?` 를 자동 생성.

### (3) DTO vs Entity — 왜 나눠요?
| | Entity (`Member`) | DTO (`MemberJoinRequestDto`) |
|---|---|---|
| 역할 | DB 테이블과 매핑 | 요청/응답 데이터 그릇 |
| 노출 | 밖에 그대로 노출하면 위험(비번 등) | 필요한 필드만 담아 주고받기 |
| 예시 | `id, userId, password, userName` | `userId, password, userName` |

### (4) 예외 공통 처리 — `@RestControllerAdvice`
- 서비스에서 문제가 생기면 `throw` 로 **던지기만** 해요.
- 컨트롤러는 그걸 잡지 않아요. 대신 `@RestControllerAdvice`가 **한 곳에서** 가로채 알맞은 상태코드(예: 409)와 JSON으로 바꿔줘요.
- 덕분에 컨트롤러는 성공 흐름에만 집중하고, 예외→응답 변환은 한 곳에 모여요.

## Step 1. 회원 엔티티 + 리포지토리 만들기

DB의 `member` 테이블과 짝이 되는 `Member` 클래스와, 그걸 저장/조회할 `MemberRepository`를 만들어요.

- `domain/entity/Member` : `@Entity`로 테이블 매핑. 필드는 `id, userId, password, userName`.
- `domain/repository/MemberRepository` : `JpaRepository`를 상속하고, **아이디 중복 검사용** 메서드를 이름 규칙으로 선언.
- `application.yaml`의 `ddl-auto`를 `update`로 두면, 엔티티 기준으로 `member` 테이블이 자동 생성돼요.

<details>
<summary>힌트 보기</summary>

```java
// domain/entity/Member.java
@Entity
@Table(name = "member")
@Getter
@Builder
@AllArgsConstructor
@NoArgsConstructor(access = AccessLevel.PROTECTED) // JPA는 기본 생성자가 필요하지만 외부 생성은 막는다
public class Member {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) // AUTO_INCREMENT
    private Long id;

    @Column(nullable = false, length = 30)
    private String userId;

    @Column(nullable = false, length = 50)
    private String password;

    @Column(nullable = false, length = 10)
    private String userName;
}
```
```java
// domain/repository/MemberRepository.java
public interface MemberRepository extends JpaRepository<Member, Long> {
    // [exists + By + UserId] → user_id 가 존재하면 true (중복 검사에 사용)
    boolean existsByUserId(String userId);
}
```
> 필드 이름은 카멜케이스(`userId`)로, DB 컬럼은 자동으로 스네이크케이스(`user_id`)로 매핑돼요.
</details>

## Step 2. 요청/응답 DTO + 매퍼 만들기

화면에서 온 JSON을 담을 **요청 DTO**, 성공 시 돌려줄 **응답 DTO**, 그리고 요청 DTO를 엔티티로 바꾸는 **매퍼**를 만들어요.

- `dto/MemberJoinRequestDto` : `userId, password, userName` (JSON이 채워지도록 `@Setter`/기본생성자 필요)
- `dto/MemberJoinResponseDto` : `url` 하나 (성공 후 이동할 주소)
- `mapper/MemberMapper` : `toEntity(요청DTO)` → `Member`

<details>
<summary>힌트 보기</summary>

```java
// dto/MemberJoinRequestDto.java  — @RequestBody(JSON)로 받으려면 setter+기본생성자가 필요
@Getter @Setter @NoArgsConstructor
public class MemberJoinRequestDto {
    private String userId;
    private String password;
    private String userName;
}
```
```java
// dto/MemberJoinResponseDto.java
@Getter @AllArgsConstructor
public class MemberJoinResponseDto {
    private String url;
}
```
```java
// mapper/MemberMapper.java
@Component
public class MemberMapper {
    public Member toEntity(MemberJoinRequestDto request) {
        return Member.builder()
                .userId(request.getUserId())
                .password(request.getPassword())   // 학습용: 평문 그대로
                .userName(request.getUserName())
                .build();
    }
}
```
> 왜 매퍼를 따로? 컨트롤러/서비스가 "DTO→엔티티 변환" 같은 잡일에 신경 안 쓰게 한곳에 모으는 거예요.
</details>

## Step 3. 서비스 — 회원가입 로직 만들기

이제 **핵심 로직**이에요. `MemberService.join()`은 (1) 아이디가 이미 있으면 예외를 던지고, (2) 없으면 엔티티로 바꿔 저장해요.

- `service/MemberService` : `@Service`, 생성자 주입(`@RequiredArgsConstructor`)으로 리포지토리·매퍼를 받음
- `join(MemberJoinRequestDto)` : 중복이면 `throw new DuplicateUserIdException(...)`, 아니면 `save`
- 저장은 DB를 바꾸므로 `@Transactional`을 붙여요.

<details>
<summary>힌트 보기</summary>

```java
// service/MemberService.java
@Service
@RequiredArgsConstructor
public class MemberService {

    private final MemberRepository memberRepository;
    private final MemberMapper memberMapper;

    @Transactional
    public void join(MemberJoinRequestDto request) {
        if (memberRepository.existsByUserId(request.getUserId())) {
            throw new DuplicateUserIdException("이미 존재하는 아이디입니다.");
        }
        Member member = memberMapper.toEntity(request);
        memberRepository.save(member);
    }
}
```
> `DuplicateUserIdException`은 Step 5에서 만들어요. (지금은 빨간 줄이 떠도 괜찮아요. Step 5까지 하면 사라져요.)
</details>

## Step 4. 컨트롤러 — 가입 API 만들기

프론트가 부르는 `POST /api/members/join` API를 만들어요. JSON 본문을 받아 서비스에 넘기고, 성공하면 이동할 주소를 응답해요.

- `controller/MemberApiController` : `@RestController`, `@RequestMapping("/api/members")`
- `join` : `@PostMapping("/join")`, `@RequestBody MemberJoinRequestDto`로 JSON 받기
- 반환: `new MemberJoinResponseDto("/members/login")`

<details>
<summary>힌트 보기</summary>

```java
// controller/MemberApiController.java
@RestController
@RequestMapping("/api/members")
@RequiredArgsConstructor
public class MemberApiController {

    private final MemberService memberService;

    @PostMapping("/join")
    public MemberJoinResponseDto join(@RequestBody MemberJoinRequestDto request) {
        memberService.join(request);
        return new MemberJoinResponseDto("/members/login");
    }
}
```
> `@RestController` = `@Controller` + `@ResponseBody`. 반환한 DTO가 **자동으로 JSON**으로 변환돼 응답 본문에 실려요.
> `@RequestBody`는 "요청 본문이 **JSON**"일 때 써요. (폼 데이터라면 `@ModelAttribute`를 써요 — 뒤 편에서 나와요.)
</details>

## Step 5. 예외 공통 처리 — 중복이면 409 응답하기

Step 3에서 던진 `DuplicateUserIdException`을, 화면이 이해할 수 있는 **409 상태코드 + 메시지 JSON**으로 바꿔요. 컨트롤러마다 try-catch 하지 않고 **한 곳**에서요.

- `exception/DuplicateUserIdException` : `RuntimeException`을 상속
- `dto/ErrorResponseDto` : `status, message`
- `exception/GlobalExceptionHandler` : `@RestControllerAdvice` + `@ExceptionHandler`로 409 응답

<details>
<summary>힌트 보기</summary>

```java
// exception/DuplicateUserIdException.java
public class DuplicateUserIdException extends RuntimeException {
    public DuplicateUserIdException(String message) {
        super(message);
    }
}
```
```java
// dto/ErrorResponseDto.java
@Getter @AllArgsConstructor
public class ErrorResponseDto {
    private int status;
    private String message;
}
```
```java
// exception/GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(DuplicateUserIdException.class)
    public ResponseEntity<ErrorResponseDto> handleDuplicateUserId(DuplicateUserIdException e) {
        return ResponseEntity
                .status(HttpStatus.CONFLICT) // 409
                .body(new ErrorResponseDto(HttpStatus.CONFLICT.value(), e.getMessage()));
    }
}
```
> 왜 409(CONFLICT)? "이미 있는 아이디와 **충돌**"이라 의미상 가장 알맞은 상태코드예요.
> 왜 `ResponseEntity`로 감싸나요? DTO만 반환하면 상태코드가 항상 200이 돼요. 409로 바꾸려면 `ResponseEntity`가 필요해요.
</details>

## Step 6. 화면에서 끝까지 확인하기 (통합 실행)

이제 조각들을 합쳐 **실제 화면**으로 돌려봐요.

1. 앱을 실행하고 브라우저에서 `GET /members/join` 접속 (회원가입 폼).
2. 새 아이디로 가입 → **로그인 페이지로 이동**하면 성공. MySQL에서 `SELECT * FROM member;` 로 저장 확인.
3. **같은 아이디로 다시** 가입 → **경고창("이미 존재하는 아이디입니다.")** 이 뜨고 이동하지 않으면 성공.

<details>
<summary>잘 안 될 때 체크리스트</summary>

- 404(주소 못 찾음): 컨트롤러 경로가 `POST /api/members/join` 이 맞는지 (`@RequestMapping` + `@PostMapping` 조합)
- 415/400(본문 못 읽음): 요청 DTO에 `@Setter`/기본 생성자가 있는지 (JSON 매핑에 필요)
- 중복인데 500이 뜸: `@RestControllerAdvice`가 붙어 있는지, `@ExceptionHandler(DuplicateUserIdException.class)`가 맞는지
- 저장이 안 됨: `application.yaml`의 `ddl-auto: update`로 `member` 테이블이 생성됐는지, DB 접속 정보가 맞는지
- 화면 메시지가 안 뜸: 실패 응답이 **409 + `{message:...}`** 형태인지 (프론트가 `message` 필드를 읽어요)
</details>
