
## I/O Stream (입출력 스트림)

- 자바에서 데이터를 입력받거나 출력할 때 사용하는, 데이터가 흐르는 통로(Stream)
- 스트림은 단방향이라서 입력용(InputStream)과 출력용(OutputStream)이 따로 존재함

#### 바이트 스트림 vs 문자 스트림

- **바이트 스트림** : 1바이트 단위로 데이터를 처리 (InputStream, OutputStream)
→ 원본 데이터를 바이트 그대로 처리
→ 이미지, 동영상, PDF, ZIP, 음악 파일 등의 바이너리 데이터
→ 대표 클래스 : `FileInputStream`, `FileOutputStream`
- **문자 스트림** : 2바이트(문자) 단위로 데이터를 처리 (Writer, Reader)
→ 바이트를 문자로 해석해서 처리
→ txt, java, csv, md 같은 텍스트 파일
→ 대표 클래스 : `FileReader` , `FileWriter` , `BufferedReader` , `BufferedWriter`
- 스트림 사용 후 반드시 `close()`로 자원을 해제해야 함
- `try-with-resources` : try()안에 만든 스트림은 try 블록 끝나면 자동으로 `close()`

#### try-with-resource 사용 예제

```java
if ( Files.notExists(todayFile) ) {
            // 방법.1 : 전통적인 방식 (직접 close)
            // finally 블록에서 스트림을 직접 닫아주어야 하므로 코드가 길고 복잡하다.
            FileOutputStream fos = null;
            try {
                fos = new FileOutputStream(todayFile.toFile());
                String content = "Hello World!";
                fos.write(content.getBytes());
            } catch (FileNotFoundException e) {
                throw new RuntimeException(e);
            } catch (IOException e) {
                throw new RuntimeException(e);
            } finally {
                if (fos != null) {
                    try {
                        fos.close();
                    } catch (IOException e) {
                        throw new RuntimeException(e);
                    }
                }
            }

            // 방법.2 -> try-with-resources
            // try(...) 괄호 안에서 선언한 자원은 try 블록이 끝나면 자동으로 close()가 호출된다.
            // 따라서 finally로 직접 닫을 필요가 없어 코드가 간결하고 안전하다.
            try ( FileOutputStream fos = new FileOutputStream(todayFile.toFile()) ) {
                String content = "Hello World!";
                fos.write(content.getBytes());
                System.out.println( today + ".txt 파일을 생성하고 내용을 썼습니다." );
            } catch (FileNotFoundException e) {
                throw new RuntimeException(e);
            } catch (IOException e) {
                throw new RuntimeException(e);
            }

        } else {
            System.out.println( today + ".txt 파일이 이미 존재합니다." );
        }
```

#### `java.nio.file` (패키지 NIO)

- Path : 파일/폴더의 경로를 표현하는 객체 
`getFileName()`, `getParent()` , `resolve(string)` (하위 경로 이어붙임)
- Paths : 문자열을 Path 객체로 만들어주는 유틸리티 클래스 
`Paths.get(string)`
- Files : Path 객체를 받아서 실제 파일/폴더 작업을 해주는 유틸리티 클래스, 유틸리티 메서드들은 모두 static
`.exists(path)` , `.notExists(path)` , `.createDirectory(path)` , `.delete(path)` , `.copy(src, dest)` , `.move(src, dest)`

#### `System.getProperty("user.home")`

- 현재 사용자의 홈 디렉토리 경로를 문자열로 반환
- JVM이 실행 환경에서 읽어온 “시스템 프로퍼티” 중 하나
- Windows면 C:\Users\사용자이름, macOS면  \Users\사용자이름, Linux면 \home\사용자 이름 같은 값임
- 즉, OS나 계정이 달라도 각자의 홈 경로를 알아서 가져옴
- 여기에 `File.separator`(OS별 경로 구분자 / 또는 \)와 “Desktop”을 붙여 바탕화면 경로를 만듦

#### `Paths.get(...)`

- 문자열로 된 경로를 Path 객체로 변환
- String은 단순한 글자 덩어리라서 파일/폴더 작업에 바로 쓸 수 없음
- Path 객체로 바꾸면 Files 클래스의 메서드(생성, 존재 확인, 삭제 등)에 인자로 넘길 수 있음
- 또 Path 객체의 `getFileName()`, `getParent()` 같은 경로 전용 기능도 사용할 수 있음
- 참고 : 자바 11 이상에서는 `Path.of(folderPath)`로 써도 동일

1. 폴더 생성
    - `Files.notExists(Path)` : 해당 경로가 존재하지 않으면 true를 반환
    - `Files.createDirectory(Path)` : 경로에 해당하는 폴더(디렉터리)를 생성
2. 파일에 내용 쓰기 - 문자를 .txt로 작성 
(**문자를** 바이트 배열로 변환 후 **바이트 스트림(**`FileOutputStream`**)으로 작성**)
    - `FileOutputStream`으로 파일에 바이트 데이터를 기록
    - 바이트 스트림은 1바이트 단위로 처리하므로, 문자열은 `getBytes()`로 byte 배열로 변환한 뒤 `write()`로 씀
    - `LocalDate.now()` : 오늘 날짜를 가져오고, 지정한 패턴("yyyy-MM-dd")의 문자열로 포맷
    - `Path.resolve(...)` : 기준 경로(myFolder) 뒤에 하위 경로를 이어 붙여 새 Path 생성

  2-1.   ⭐파일에 내용 쓰기 - 문자를 .txt로 작성 
           **(문자를 문자열 스트림(`FileWriter`)으로 작성)**

- exam2()는 바이트 스트림(`FileOutputStream`)이라 문자열을 getBytes()로 byte 배열로 바꿔서 써야 했다.
- 문자 스트림(`FileWriter`)은 문자(char) 단위로 처리하므로 String을 그대로 write()에 넘길 수 있다. (byte 변환 불필요)
- 그래서 텍스트 파일을 다룰 때는 문자 스트림이 더 간편하다.

  2-2.  파일에 내용 쓰기 - QR코드 .png 만들기 
           (문자열 → 행렬 → 이미지 파일을 바이트 데이터로 기록)

- 바이트 스트림은 직접 드러나 있지 않고, `MatrixToImageWriter.writeToPath()` 가 PNG 파일을 저장할 때 사용되는 파일 출력 방식으로 볼 수 있다.
- QR코드는 이미지(PNG)이므로 문자가 아닌 "바이너리 데이터"다. → 바이트 스트림이 적합하다.
- 2-1.처럼 텍스트가 아니라 이미지라서, 문자 스트림으로는 다룰 수 없는 대표 예시다.
- ZXing 라이브러리로 문자열을 QR코드 행렬(BitMatrix)로 만든 뒤 PNG 파일로 저장한다.
1)   `QRCodeWriter.encode(...)` : 문자열을 QR코드 행렬(BitMatrix)로 변환
-  인자 : 내용, 포맷(QR_CODE), 가로 크기, 세로 크기(픽셀)
2)   `MatrixToImageWriter.writeToPath(...)` : 행렬을 실제 PNG 이미지 파일로 저장
- 인자 : 행렬, 포맷(”PNG”), path 
- 내부적으로 ImageIO(바이트 스트림)를 이용해 바이너리로 기록한다.
1. 파일 내용 읽기 ↔ 2.의 짝
(**바이트 스트림(**`FileInputStream`**)으로 읽은 바이트 배열을 문자열로 변환**)
    - `FileInputStream`으로 파일을 1바이트 단위로 읽어 들인다.
    - 읽은 byte 배열을 new String(...)으로 다시 문자열로 변환해야 사람이 읽을 수 있다.

  3-2.  ⭐파일 내용 읽기 문자 스트림 ↔ 2-1.의 짝 
           (**문자열 스트림(`FileReader`)으로 문자열 읽기)**

- `FileReader`로 파일을 문자(char) 단위로 읽는다.
- 바이트 스트림과 달리 byte → String 변환이 필요 없다.
- 보통 `BufferedReader`로 감싸 한 줄씩(readLine) 편하게 읽는다.

  3-1.  QR코드로 이미지 읽기 ↔ 2-2.의 짝
           (이미지 파일을 바이트 데이터로 읽고 → 비트맵 → 문자열)

- 저장된 PNG를 `BufferedImage`로 읽고, ZXing으로 다시 문자열을 추출(decode)한다.
- 쓰기(encode)의 반대 과정이다: 문자열 → 이미지 였다면, 여기선 이미지 → 문자열
1) `ImageIO.read(...)` : PNG 파일을 BufferedImage(이미지 객체)로 읽어온다.
2) 이미지를 ZXing이 해석할 수 있는 형태(BinaryBitmap)로 변환
3) `MultiFormatReader.decode(...)` : 비트맵에서 QR코드 내용을 추출

#### ZXing 라이브러리 (지브라 크로싱)

- QR코드나 바코드를 만들고 읽을 수 있게 해주는 외부 라이브러리
- `MatrixToImageWriter` : `BitMatrix`를 이미지, 파일, 스트림으로 쓸 수 있게 해주는 클래스

build.gradle.kts의 dependencies에 추가

- `implementation("그룹ID:라이브러리이름:버전")` : Gradle이 빌드할 때 Maven 저장소에서 해당 라이브러리를 다운로드하고, 코드에서 `import`해서 쓸 수 있게 해줌

```java
// ZXing QR코드 생성 라이브러리를 Gradle 프로젝트에 의존성으로 추가

// core : QR코드/바코드 데이터를 만들고 해석하는 핵심 로직
// QRCodeWriter, BarcodeFormat, BitMatrix, WriterException
implementation("com.google.zxing:core:3.5.3") 
// javase : BitMatrix를 PNG 이미지 파일로 저장하는 등 Java 환경에서 이미지 입출력을 도와주는 기능
// MatrixToImageWriter
implementation("com.google.zxing:javase:3.5.3")
```

#### Gradle :

- 자바 프로젝트에서 사용하는 **빌드 도구**
- 프로젝트 외부 라이브러리 다운로드/컴파일/테스트/빌드/실행를 자동으로 해주는 도구
- Gradle이 없다면, ZXing 라이브러리를 직접 다운로드해서 `.jar` 파일을 프로젝트에 넣고, 클래스패스 설정까지 직접 해야 함

#### build.gradle.kts :

- Gradle에게 “이 프로젝트는 어떤 라이브러리를 쓰고, 어떻게 빌드해야 하는지” 알려주는 설정 파일

#### Maven :

- Gradle처럼 자바 프로젝트의 라이브러리 관리, 컴파일, 테스트, 빌드를 자동으로 해주는 빌드 도구이고, 설정 파일로 `pom.xml`을 사용

#### Gradle vs Maven 차이

Maven
→ pom.xml 사용
→ 규칙이 정해져 있음
→ 비교적 전통적이고 안정적인 방식

Gradle
→ build.gradle 또는 build.gradle.kts 사용
→ 더 유연함
→ 요즘 Android, Spring 프로젝트에서도 많이 사용
