# 프로세스

프로세스는 **실행 중인 프로그램**이다.

프로그램은 실행되기 전까지는 보조기억장치에 저장된 데이터 덩어리일 뿐이다.  
하지만 프로그램이 메모리에 적재되어 실행되는 순간, 그 프로그램은 **프로세스**가 된다.

프로세스는 실행 위치에 따라 다음과 같이 나눌 수 있다.

- **포그라운드 프로세스**  
  사용자가 볼 수 있는 공간에서 실행되는 프로세스이다.  
  예를 들어 사용자가 직접 실행한 웹 브라우저, 문서 편집기 등이 있다.

- **백그라운드 프로세스**  
  사용자가 직접 볼 수 없는 공간에서 실행되는 프로세스이다.

- **데몬 또는 서비스**  
  백그라운드 프로세스 중에서 사용자와 직접 상호작용하지 않고, 정해진 작업을 계속 수행하는 프로세스이다.

---

# 프로세스 제어 블록

운영체제는 여러 프로세스가 CPU를 나누어 사용할 수 있도록 관리한다.

프로세스들은 CPU를 한 번에 계속 사용하는 것이 아니라, 정해진 시간만큼만 CPU를 사용한다.  
자신의 차례가 되면 CPU를 사용하고, 시간이 끝났음을 알리는 **타이머 인터럽트**가 발생하면 CPU 사용을 멈춘 뒤 다음 차례를 기다린다.

운영체제는 이렇게 빠르게 번갈아 실행되는 프로세스들의 실행 순서를 관리하고, CPU와 메모리 같은 자원을 배분한다.  
이때 운영체제가 사용하는 자료구조가 **프로세스 제어 블록**, 즉 **PCB**이다.

PCB는 프로세스와 관련된 중요한 정보를 저장하는 자료구조이며, **커널 영역**에 생성된다.

## PCB에 저장되는 정보

PCB에는 다음과 같은 정보가 저장된다.

- **프로세스 ID**
- **레지스터 값**
- **프로세스 상태**
- **CPU 스케줄링 정보**
- **메모리 관리 정보**
- **사용한 파일 정보**
- **입출력 장치 정보**

즉, PCB는 운영체제가 프로세스를 관리하기 위해 필요한 정보를 담고 있는 자료구조이다.

# 문맥 교환

문맥이란 하나의 프로세스 실행을 나중에 다시 이어서 수행하기 위해 기억해야 할 정보이다.

예를 들어 프로세스가 어디까지 실행되었는지, 레지스터에는 어떤 값이 들어 있었는지 등의 정보가 문맥에 해당한다.

**문맥 교환**은 현재 실행 중인 프로세스의 문맥을 PCB에 저장하고, 다음에 실행할 프로세스의 문맥을 PCB에서 복구하는 작업이다.

즉, 문맥 교환은 다음과 같은 과정으로 이루어진다.

1. 현재 실행 중인 프로세스의 상태를 PCB에 저장한다.
2. 다음에 실행할 프로세스의 상태를 PCB에서 불러온다.
3. CPU가 새로운 프로세스를 실행한다.

이 과정을 통해 여러 프로세스가 빠르게 번갈아 실행될 수 있다.

다만 문맥 교환이 너무 자주 발생하면, 실제 작업보다 프로세스를 바꾸는 데 시간이 많이 소요될 수 있다.  
이처럼 추가로 발생하는 부담을 **오버헤드**라고 한다.

# 프로세스의 메모리 영역

프로세스가 생성되면 커널 영역에는 해당 프로세스의 PCB가 생성되고, 사용자 영역에는 프로세스가 실행되기 위한 내용이 저장된다.

사용자 영역의 프로세스 메모리는 크게 다음 네 영역으로 나뉜다.

## 코드 영역

코드 영역은 CPU가 실행할 명령어가 저장되는 공간이다.

기계어로 이루어진 프로그램의 코드가 저장되며, 실행 중에 수정되면 안 되기 때문에 일반적으로 **읽기 전용**으로 사용된다.

## 데이터 영역

데이터 영역은 프로그램이 실행되는 동안 유지되어야 하는 데이터가 저장되는 공간이다.

대표적으로 **전역 변수**와 **정적 변수**가 데이터 영역에 저장된다.  
이 데이터들은 프로그램이 실행되는 동안 계속 유지된다.

## 힙 영역

힙 영역은 프로그래머가 직접 메모리를 할당할 수 있는 공간이다.

프로그램 실행 중 필요한 만큼 메모리를 할당하고 사용할 수 있다.  
하지만 할당한 메모리는 사용이 끝나면 반드시 반환해야 한다.

할당한 메모리를 반환하지 않으면, 해당 공간이 계속 메모리에 남아 낭비가 발생한다.  
이러한 문제를 **메모리 누수**라고 한다.

## 스택 영역

스택 영역은 데이터를 일시적으로 저장하는 공간이다.

함수를 호출할 때 생성되는 **매개변수**, **지역 변수**, **복귀 주소** 등이 저장된다.  
함수 실행이 끝나면 해당 함수에서 사용한 스택 영역의 데이터는 사라진다.

---

# 정적 할당 영역과 동적 할당 영역

프로세스의 메모리 영역은 크기의 변화 여부에 따라 다음과 같이 나눌 수 있다.

| 구분 | 영역 | 특징 |
|---|---|---|
| 정적 할당 영역 | 코드 영역, 데이터 영역 | 프로그램 실행 중 크기가 거의 변하지 않는다. |
| 동적 할당 영역 | 힙 영역, 스택 영역 | 프로그램 실행 중 크기가 변할 수 있다. |

코드 영역과 데이터 영역은 프로그램 실행 중 크기가 고정되어 있기 때문에 **정적 할당 영역**이라고 한다.

반면 힙 영역과 스택 영역은 프로그램 실행 과정에서 크기가 변할 수 있기 때문에 **동적 할당 영역**이라고 한다.

# 스레드

스레드는 **프로세스를 구성하는 실행의 흐름 단위**이다.

하나의 프로세스는 하나 이상의 스레드를 가질 수 있다.  
스레드를 이용하면 하나의 프로세스 안에서 여러 작업을 동시에 실행할 수 있다.

예를 들어 하나의 프로그램 안에서 화면 출력, 사용자 입력 처리, 파일 다운로드를 각각 다른 스레드가 담당할 수 있다.

스레드는 실행에 필요한 최소한의 정보를 가진다.

스레드는 다음과 같은 정보를 가진다.

- **스레드 ID**
- **프로그램 카운터**
- **레지스터 값**
- **스택**

## 프로그램 카운터

프로그램 카운터는 다음에 실행할 명령어가 메모리의 어디에 있는지를 가리키는 값이다.

CPU 안에 있는 레지스터의 한 종류이며, CPU가 명령어를 순서대로 실행할 수 있도록 도와준다.

---

# 프로세스와 스레드의 관계

프로세스는 실행 중인 프로그램이며, 실행에 필요한 자원을 담고 있는 공간에 가깝다.

스레드는 그 프로세스 안에서 실제로 명령어를 실행하는 실행 단위이다.

즉, 프로세스는 실행 환경이고, 스레드는 그 안에서 실제 작업을 수행하는 흐름이다.

쉽게 정리하면 다음과 같다.

- **프로세스**  
  실행 중인 프로그램이다.  
  코드, 데이터, 메모리 같은 자원을 담고 있는 실행 환경이다.

- **스레드**  
  프로세스 안에서 실제로 명령어를 실행하는 단위이다.  
  CPU가 실제로 실행하는 작업의 흐름이라고 볼 수 있다.

프로세스가 해야 할 일을 실제로 수행하는 것은 스레드이다.

---

# 멀티프로세스와 멀티스레드

컴퓨터는 여러 프로세스를 동시에 실행할 수 있고, 하나의 프로세스 안에서도 여러 스레드를 실행할 수 있다.

## 멀티프로세스

멀티프로세스는 여러 개의 프로세스를 동시에 실행하는 방식이다.

예를 들어 같은 프로그램을 여러 번 실행하면, 각각 별도의 프로세스가 생성된다.

멀티프로세스에서는 각 프로세스가 기본적으로 독립된 메모리 공간과 자원을 가진다.

## 멀티스레드

멀티스레드는 하나의 프로세스 안에서 여러 개의 스레드를 동시에 실행하는 방식이다.

프로세스가 해야 할 일을 하나의 스레드가 순서대로 처리하는 대신, 여러 스레드가 나누어 처리한다.

멀티스레드를 사용하면 하나의 프로세스 안에서 여러 작업을 동시에 수행할 수 있다.

---

# 멀티프로세스와 멀티스레드의 차이

예를 들어 `"hello"`를 출력하는 프로그램이 있다고 하자.

이 프로그램을 세 개의 프로세스로 실행하면 `"hello"`가 세 번 출력된다.  
또한 하나의 프로세스 안에서 `"hello"`를 출력하는 스레드를 세 개 만들어도 `"hello"`는 세 번 출력된다.

겉으로 보기에는 결과가 비슷해 보일 수 있다.  
하지만 내부 동작 방식에는 큰 차이가 있다.

## 자원 공유 방식의 차이

프로세스끼리는 기본적으로 자원을 공유하지 않는다.

반면 같은 프로세스 안의 스레드들은 프로세스의 자원을 공유한다.  
예를 들어 코드 영역, 데이터 영역, 힙 영역 등을 함께 사용할 수 있다.

## 메모리 사용의 차이

멀티프로세스 방식에서는 같은 프로그램을 여러 번 실행하면 각 프로세스마다 자원이 따로 생성된다.  
따라서 동일한 내용이 메모리에 중복으로 존재할 수 있다.

반면 멀티스레드 방식에서는 하나의 프로세스 안에서 여러 스레드가 자원을 공유한다.  
따라서 멀티프로세스보다 메모리를 더 효율적으로 사용할 수 있다.

## 안정성의 차이

멀티프로세스 환경에서는 하나의 프로세스에 문제가 생겨도 다른 프로세스에 미치는 영향이 비교적 적다.

하지만 멀티스레드 환경에서는 여러 스레드가 같은 프로세스의 자원을 공유하기 때문에, 하나의 스레드에 문제가 생기면 프로세스 전체에 문제가 생길 수 있다.

## 통신의 차이

프로세스끼리는 자원을 공유하지 않기 때문에 서로 데이터를 주고받기 위해 별도의 통신 방법이 필요하다.

반면 스레드끼리는 같은 프로세스의 자원을 공유하기 때문에 협력과 통신이 비교적 쉽다.

---

# 멀티프로세스와 멀티스레드 비교

| 구분 | 멀티프로세스 | 멀티스레드 |
|---|---|---|
| 실행 단위 | 여러 프로세스 | 하나의 프로세스 안의 여러 스레드 |
| 자원 공유 | 기본적으로 자원을 공유하지 않음 | 같은 프로세스의 자원을 공유함 |
| 메모리 사용 | 자원이 중복되어 메모리 사용량이 커질 수 있음 | 자원을 공유하므로 메모리를 효율적으로 사용함 |
| 안정성 | 하나의 프로세스 문제가 다른 프로세스에 미치는 영향이 적음 | 하나의 스레드 문제가 프로세스 전체에 영향을 줄 수 있음 |
| 통신 | 프로세스 간 통신이 비교적 복잡함 | 스레드 간 통신이 비교적 쉬움 |
| 예시 | 여러 개의 프로그램을 각각 실행 | 하나의 프로그램 안에서 여러 작업을 동시에 처리 |

# 한 줄 정리

프로세스는 실행 중인 프로그램이고, 스레드는 프로세스 안에서 실제로 작업을 수행하는 실행 흐름이다.

멀티프로세스는 여러 프로세스를 동시에 실행하는 방식이고, 멀티스레드는 하나의 프로세스 안에서 여러 스레드가 자원을 공유하며 동시에 실행되는 방식이다.
