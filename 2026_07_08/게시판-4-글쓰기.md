# 글쓰기 만들기 (파일 업로드 + 다운로드)

> 게시판에 **글을 쓰고 첨부파일을 올리는** 기능을 만들어요. 여태 다룬 건 텍스트(JSON)뿐이었는데, 이번엔 **파일이 섞인 요청(multipart)** 을 받아 디스크에 저장하고, 반대로 저장한 파일을 **내려받는(download)** 흐름까지 익혀요.
> **아래 Step을 순서대로 따라가면, 글에 파일을 첨부해 저장하고 상세 화면에서 다시 내려받는 글쓰기가 완성됩니다.**
>
> 각 Step의 **힌트는 접혀 있어요.** "이건 JSON일까 폼(multipart)일까?", "이 응답은 브라우저가 열까 저장할까?"를 먼저 떠올리고, 막히면 "힌트 보기"를 펼치세요.
>
> 이번 편에서는 파일 처리 로직을 **`BoardService` 안에 그대로** 둬요. 다음 편(5편)에서 이걸 `FileService`로 **분리(리팩터링)** 할 거예요. "일단 한 곳에 만든 뒤 나중에 나눈다"는 흐름 자체가 학습 포인트예요.

---

## 0. 먼저 알아둘 점

- 이 과제는 **1~3편을 마쳤다는 전제**예요. 아래는 이미 있어요:
  - `Board` 엔티티와 `BoardRepository`(JpaRepository) — 3편에서 만든 것
  - `application.yaml`의 DB 접속 정보, JPA 설정
  - **프론트 화면/스크립트**(`board-write.html`·`boardWrite.js`, `board-detail.html`·`boardDetail.js`) — 여러분은 화면을 건드리지 않아요
  - 글쓰기 폼을 보여주는 **뷰 컨트롤러**(`GET /write`)
- **여러분이 작성할 것**: 업로드 설정 · 쓰기 요청 DTO · 파일 저장 로직 · 쓰기 API · 다운로드 API.
- 파일은 프로젝트 루트의 `./uploads` 폴더에 저장돼요. (폴더가 없으면 코드가 만들어 줘요.)

> 프론트가 이미 정해둔 **약속(계약)** 이에요. 여러분의 백엔드는 여기에 맞춰야 해요:
> - 글쓰기 요청: `boardWrite.js`가 **`multipart/form-data`(FormData)** 로 `POST /api/boards` 를 호출. 폼 필드는 `title`, `content`, `userId`(hidden), `file`(선택)
> - 성공 응답: 상태 **200**(본문 없음) → JS가 목록으로 이동
> - 다운로드: `boardDetail.js`가 `<a href="/api/boards/file/download/{fileName}">` 링크로 GET 호출 → 브라우저가 **다운로드**

## 1. 무엇을 만드나요?

글쓰기 폼에 제목·내용을 쓰고 파일을 첨부해 **작성**을 누르면 → 파일은 **디스크(`./uploads`)에 저장**되고, 그 **경로가 글과 함께 DB에 저장**돼요. 상세 화면에서는 첨부파일 이름을 눌러 **내려받을** 수 있어요.

| 주소(메서드) | 하는 일 | 계층 |
|------|------|------|
| `GET /write` | 글쓰기 폼 보여주기 | 뷰 컨트롤러 (**제공**) |
| `POST /api/boards` | 글 저장 + 첨부파일 저장 | **여러분이 만들 API** |
| `GET /api/boards/file/download/{fileName}` | 첨부파일 내려주기 | **여러분이 만들 API** |

**완성되면 이렇게 동작해요**
```
[글쓰기]
  제목: 공지 / 내용: 안녕하세요 / 파일: 이력서.pdf  → [작성]
  → uploads/ 에 "e1f2...-이력서.pdf" 로 저장
  → board 테이블에 title, content, file_path 저장 → 목록으로 이동

[다운로드]
  상세 화면에서 "이력서.pdf" 링크 클릭
  → 브라우저가 원래 이름("이력서.pdf")으로 파일을 저장(다운로드)
```

## 3. 핵심 개념

### (1) `@RequestBody`(JSON) vs `@ModelAttribute`(파일 섞인 폼)

지금까지(회원가입 등)는 본문이 **JSON**이라 `@RequestBody`를 썼어요. 그런데 **파일은 JSON에 실을 수 없어요.** 그래서 파일이 섞인 요청은 `multipart/form-data`로 오고, 이건 `@ModelAttribute`로 받아요.

| | `@RequestBody` | `@ModelAttribute` |
|---|---|---|
| 언제 | 본문이 **JSON**일 때 | **폼 데이터(multipart)** 일 때 |
| 파일 담기 | ❌ (JSON엔 파일 못 실음) | `MultipartFile`을 **필드로** |
| 채우는 방식 | Jackson이 JSON 파싱 | **기본 생성자 + setter**(JavaBean) |

> 그래서 `MultipartFile`도 특별한 게 아니라 **DTO의 한 필드**예요. 폼의 `name`과 DTO **필드 이름이 같아야** 자동으로 채워져요.

### (2) MultipartFile과 `transferTo` — 임시 파일이 최종 위치로

- 브라우저가 파일을 올리면, **서블릿 컨테이너(내장 톰캣)** 가 먼저 **임시 위치**(시스템 임시 폴더)에 저장해 둬요. 그게 `MultipartFile`로 컨트롤러에 전달돼요.
- 이 임시 파일은 **요청이 끝나면 사라져요.** 그래서 처리 중인 지금 `file.transferTo(최종경로)`로 **옮겨 둬야** 해요.
- 임시 위치와 최종 위치가 같은 디스크면 `transferTo`는 복사가 아니라 **이름만 바꾸는 이동(rename)** 이라 가벼워요.

### (3) 파일 저장, 이렇게 하면 안전해요 (옛 방식과 비교)
흔한 옛 예시 `byte[] bytes = file.getBytes(); Files.write(폴더+파일명, bytes);` 와 비교하면:

| 관점 | `getBytes()` + `Files.write` | 우리 방식(`transferTo` + UUID) |
|---|---|---|
| 메모리 | 파일 **전체**를 힙에 올림 → 큰 파일/동시 업로드 시 **OOM 위험** | 통째로 안 올리고 이동 → 가벼움 |
| 파일명 | 원본 이름 그대로 → 같은 이름이면 **덮어씀** | **UUID**를 앞에 붙여 항상 고유 |
| 경로 | `"폴더"+"파일명"` 문자열 결합 → 구분자/폴더 없음에 취약 | `new File(dir, name)` + `mkdirs()`로 안전 |

### (4) 다운로드의 핵심 — 두 헤더의 역할이 다르다
- `Content-Type: application/octet-stream` → "무슨 파일인지 특정 안 한 순수 바이너리"라는 **힌트(유도)**. 다운로드를 **확정하진 못해요.**
- `Content-Disposition: attachment; filename*=...` → "열지 말고 **저장(다운로드)** 하라"는 **확실한 지시(결정타)** + 저장될 파일명 지정.
- 그래서 다운로드 API는 헤더를 직접 세팅해야 하고, 그러려면 **`ResponseEntity`로 감싸야** 해요.

## Step 1. 파일 업로드 설정하기

파일을 어디에 저장할지(`file.upload-dir`)와, 업로드 최대 크기를 `application.yaml`에 설정해요. 기본 최대 크기(1MB)는 이미지에도 쉽게 걸리므로 늘려줘요.

- `file.upload-dir` : 저장 폴더 (예: `./uploads`)
- `spring.servlet.multipart.max-file-size` / `max-request-size` : 예) `10MB`

<details>
<summary>힌트 보기</summary>

```yaml
# application.yaml
file:
  upload-dir: ./uploads          # 첨부파일을 저장할 폴더 (프로젝트 루트 기준)

spring:
  servlet:
    multipart:
      max-file-size: 10MB        # 파일 1개 최대 크기 (기본 1MB라 쉽게 초과됨)
      max-request-size: 10MB     # 요청 전체 최대 크기
```
> `file.upload-dir` 값은 Step 3에서 `@Value("${file.upload-dir}")`로 주입받아 써요.
</details>

## Step 2. 글쓰기 요청 DTO 만들기 (`@ModelAttribute` + `MultipartFile`)

폼으로 온 값(`title, content, userId`)과 **파일**을 **하나의 DTO**에 담아요. 파일이 섞인 폼이므로 `@ModelAttribute`로 받을 거고, 그러려면 **setter + 기본 생성자**가 필요해요.

- `dto/BoardWriteRequestDto` : `title`, `content`, `userId`, `MultipartFile file`
- 필드 이름은 폼의 `name`과 **똑같이** (`title`, `content`, `userId`, `file`)

<details>
<summary>힌트 보기</summary>

```java
// dto/BoardWriteRequestDto.java  — @ModelAttribute로 받으려면 setter+기본생성자가 필요(JavaBean)
@Getter @Setter @NoArgsConstructor
public class BoardWriteRequestDto {
    private String title;
    private String content;
    private String userId;       // 화면 hidden input(name="userId")의 로그인 아이디
    private MultipartFile file;  // 첨부파일 (선택) — 안 고르면 null
}
```
> 파일도 그냥 **필드 하나**예요. `@RequestPart`를 따로 안 써도 `@ModelAttribute`가 폼 필드와 파일을 함께 채워줘요.
> 필드 이름이 폼 `name`과 하나라도 다르면 그 값은 `null`로 들어와요.
</details>

## Step 3. 파일 저장 로직 만들기 (`storeFile`)

업로드된 파일을 `./uploads`에 저장하고 **저장 경로를 반환**하는 `storeFile`을 만들어요. 그리고 `saveArticle`이 그 경로를 글과 함께 저장하게 해요.

- `service/BoardService` : `@Value("${file.upload-dir}") String uploadDir` 주입
- `storeFile(MultipartFile)` : 비었으면 `null`; 아니면 폴더 생성 → `UUID_원본이름` → `transferTo` → 경로 반환
- `saveArticle(userId, title, content, file)` : `storeFile`로 경로를 얻어 `Board`를 만들어 저장 (`@Transactional`)

<details>
<summary>힌트 보기</summary>

```java
// service/BoardService.java (일부)
@Value("${file.upload-dir}")
private String uploadDir;

@Transactional
public void saveArticle(String userId, String title, String content, MultipartFile file) {
    String filePath = storeFile(file);   // 첨부파일 없으면 null 반환

    Board board = Board.builder()
            .userId(userId)
            .title(title)
            .content(content)
            .filePath(filePath)
            .created(LocalDateTime.now())
            .build();

    boardRepository.save(board);
}

private String storeFile(MultipartFile file) {
    if (file == null || file.isEmpty()) return null;   // 첨부 안 함

    try {
        File dir = new File(uploadDir).getAbsoluteFile();
        if (!dir.exists()) dir.mkdirs();               // 폴더 없으면 생성

        // 같은 이름이 덮어써지지 않도록 UUID를 앞에 붙여 고유하게
        String storedName = UUID.randomUUID() + "_" + file.getOriginalFilename();
        File dest = new File(dir, storedName);          // 구분자 안전하게 결합

        file.transferTo(dest);                          // 임시 파일 → 최종 위치로 이동
        return dest.getPath();
    } catch (IOException e) {
        throw new IllegalStateException("파일 저장에 실패했습니다.", e);
    }
}
```
> `transferTo` 전의 "임시 파일"은 톰캣이 시스템 임시 폴더에 만들어 둔 거예요. 요청이 끝나면 사라지니 지금 옮겨야 해요.
</details>

## Step 4. 글쓰기 컨트롤러 만들기 (`@ModelAttribute`, `void`)

프론트가 부르는 `POST /api/boards`를 만들어요. 파일이 섞인 폼이라 `@ModelAttribute`로 DTO를 받고, 서비스에 넘겨요.

- `controller/BoardApiController` : `@RestController`, `@RequestMapping("/api/boards")`
- `saveArticle` : `@PostMapping`, `@ModelAttribute BoardWriteRequestDto`
- 반환은 **`void`** — 돌려줄 본문도, 세팅할 헤더도 없으니 감쌀 필요 없어요(스프링이 200 OK로 응답).

<details>
<summary>힌트 보기</summary>

```java
// controller/BoardApiController.java (일부)
@PostMapping
public void saveArticle(@ModelAttribute BoardWriteRequestDto request) {
    boardService.saveArticle(
            request.getUserId(),
            request.getTitle(),
            request.getContent(),
            request.getFile()
    );
}
```
> **`ResponseEntity`로 감싸지 않았어요.** 상태코드 200이면 충분하고 헤더도 본문도 없으니 `void`가 맞아요.
> 반대로 다음 Step의 다운로드는 헤더를 직접 세팅해야 해서 `ResponseEntity`로 감싸요. (역할이 있을 때만 감싸는 거예요.)
</details>

## Step 5. 다운로드 API 만들기 (`Resource` + `Content-Disposition`)

저장한 파일을 내려주는 `GET /api/boards/file/download/{fileName}`을 만들어요. 파일을 `Resource`로 읽어, 브라우저가 **저장(다운로드)** 하도록 헤더를 세팅해 응답해요.

- `service/BoardService.downloadFile(fileName)` : `uploads`에서 파일을 찾아 `Resource`로 반환 (없으면 404 예외)
- `controller` : 한글 파일명을 URL 인코딩 → `octet-stream` + `attachment` 헤더 → `ResponseEntity`로 응답

<details>
<summary>힌트 보기</summary>

```java
// service/BoardService.java (일부) — 파일을 Resource로 반환
public Resource downloadFile(String fileName) {
    try {
        File file = new File(new File(uploadDir).getAbsoluteFile(), fileName);
        Resource resource = new UrlResource(file.toURI());
        if (!resource.exists() || !resource.isReadable()) {
            throw new BoardNotFoundException("파일을 찾을 수 없습니다. fileName=" + fileName);
        }
        return resource;
    } catch (MalformedURLException e) {
        throw new IllegalStateException("파일 경로가 잘못되었습니다.", e);
    }
}
```
```java
// controller/BoardApiController.java (일부)
@GetMapping("/file/download/{fileName}")
public ResponseEntity<Resource> downloadFile(@PathVariable String fileName) {
    Resource resource = boardService.downloadFile(fileName);

    // 한글/공백은 헤더에 그대로 못 담으니 URL 인코딩 ( '+' → %20 )
    String encoded = URLEncoder.encode(resource.getFilename(), StandardCharsets.UTF_8)
            .replaceAll("\\+", "%20");

    return ResponseEntity.ok()
            .contentType(MediaType.APPLICATION_OCTET_STREAM)                     // "순수 바이너리" 힌트(유도)
            .header(HttpHeaders.CONTENT_DISPOSITION,
                    "attachment; filename*=UTF-8''" + encoded)                   // "저장하라" 지시(결정타)
            .body(resource);
}
```
> 왜 `byte[]`가 아니라 `Resource`? 파일을 통째로 메모리에 올리면 큰 파일에서 OOM 위험이 있어요. `Resource`는 스트림으로 흘려보내 가벼워요.
> `octet-stream`은 저장을 **유도**하는 힌트일 뿐, 실제 다운로드를 **확정**하는 건 `Content-Disposition: attachment`예요.
</details>

## Step 6. 화면에서 끝까지 확인하기 (통합 실행)

이제 조각들을 합쳐 **실제 화면**으로 돌려봐요.

1. 앱 실행 후 `GET /write` 접속 → 제목·내용 입력, 파일 첨부 → **작성**.
2. 목록으로 이동하면 성공. 프로젝트의 `./uploads` 폴더에 `UUID_원본이름` 파일이 생겼는지, `SELECT * FROM board;` 로 `file_path`가 저장됐는지 확인.
3. 상세 화면에서 첨부파일 링크 클릭 → **원래 이름으로 다운로드**되면 성공.

<details>
<summary>잘 안 될 때 체크리스트</summary>

- 400/저장 실패: 폼 `name`과 DTO 필드명이 같은지(`title/content/userId/file`), DTO에 `@Setter`/기본생성자가 있는지
- 파일이 저장 안 됨: `application.yaml`에 `file.upload-dir`이 있는지, 서비스에 `@Value("${file.upload-dir}")`로 주입했는지
- 413(너무 큼): `max-file-size`/`max-request-size`를 늘렸는지
- 다운로드가 "저장"이 아니라 브라우저에서 열림: `Content-Disposition: attachment`가 빠졌는지
- 한글 파일명이 깨짐: 파일명을 `URLEncoder`로 인코딩하고 `filename*=UTF-8''` 형식으로 넣었는지
- 404: 다운로드 경로가 `/api/boards/file/download/{fileName}` 이 맞는지
</details>
