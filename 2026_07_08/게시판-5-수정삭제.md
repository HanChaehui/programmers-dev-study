# 게시글 수정·삭제 (dirty checking)

> 게시판 CRUD의 마지막 두 조각인 **수정(U)** 과 **삭제(D)** 를 완성해요. 그러면서 JPA의 **변경 감지(dirty checking)** 로 `save()` 없이 수정하는 마법, 파일이 딸린 글을 **안전하게 지우는 순서**, 그리고 커진 서비스를 **FileService로 나누는 리팩터링**까지 익혀요.
> **아래 Step을 순서대로 따라가면, 목록·상세·작성에 더해 수정·삭제까지 되는 게시판이 완성되고, 서비스 코드가 한결 깔끔해집니다.**
>
> 각 Step의 **힌트는 접혀 있어요.** "이건 어느 계층 책임이지?"를 먼저 떠올리고, 막히면 "힌트 보기"를 펼치세요.
>
> 이번 편은 **1~4편(회원가입·로그인·읽기·글쓰기/파일)이 이미 됐다는 전제**예요. 특히 **파일 업로드(`storeFile`)** 와 **`Board` 엔티티**가 있어야 해요.

---

## 0. 먼저 알아둘 점

- 앞 편들에서 만든 것들을 **그대로 이어서** 써요. 아래는 제공돼요:
  - **프론트 화면·JS**: `board-detail.html` / `boardDetail.js`(삭제 버튼), `board-update.html` / `boardUpdate.js`(수정 폼)
  - 수정 화면으로 가는 링크: 상세의 `editArticle()` → `/update/{id}` 이동
  - 수정 폼을 보여주는 **뷰 컨트롤러**(`GET /update/{id}`)
- **여러분이 작성할 것**: `deleteArticle`·`updateArticle` **서비스/컨트롤러/DTO**, `Board.update` **엔티티 메서드**, 그리고 파일 로직을 옮길 **`FileService`**.

> 프론트가 이미 정해둔 **약속(계약)** 이에요. 백엔드는 여기에 맞춰야 해요:
> - **삭제**: `DELETE /api/boards/{id}` + **JSON 본문** `{ "filePath": "..." }`
> - **수정**: `PUT /api/boards/{id}` + **multipart/form-data**(`title`, `content`, `fileFlag`, `file`)
>   - `fileFlag` = 사용자가 첨부파일을 **건드렸는지**(교체/제거) 여부. 안 건드리면 `false`

## 1. 무엇을 만드나요?

기존 게시판에 **수정**과 **삭제**를 더해요. 둘 다 첨부파일까지 함께 다루는 게 포인트예요.

| 주소(메서드) | 하는 일 | 계층 |
|------|------|------|
| `GET /update/{id}` | 수정 폼 보여주기(기존 값 채움) | 뷰 컨트롤러 (**제공**) |
| `PUT /api/boards/{id}` | 글 수정(+파일 유지/교체/제거) | **여러분이 만들 API** |
| `DELETE /api/boards/{id}` | 글 삭제(+첨부파일 삭제) | **여러분이 만들 API** |

**완성되면 이렇게 동작해요**
```
[수정]
  상세 → [수정] → /update/3 (제목·내용 채워짐)
  내용 고치고 파일 교체 → [수정하기] → 목록으로 이동 (기존 파일은 지워지고 새 파일로 교체)

[삭제]
  상세 → [삭제] → "삭제되었습니다" → 목록으로 이동 (board 행 + 첨부파일 모두 사라짐)
```

## 2. 학습 목표

| 개념 | 어디서 배우나 |
|------|------|
| `deleteById`로 삭제 + 파일까지 지우는 **순서** | Step 1 (`deleteArticle`) |
| `@RequestBody`(JSON) DTO — setter가 없으면 값이 `null`이 되는 함정 | Step 1 (`BoardDeleteRequestDto`) |
| setter 대신 **의미 있는 메서드** + JPA **변경 감지(dirty checking)** | Step 2 (`Board.update`) |
| `fileFlag`로 파일 **유지/교체/제거** 3경우 처리 | Step 3 (`updateArticle`) |
| `PUT` + `@ModelAttribute`(id는 경로로) | Step 4 (`BoardApiController`) |
| **단일 책임 원칙(SRP)** — 파일 로직을 `FileService`로 분리·위임 | Step 5 (`FileService`) |
| 클래스 레벨 `@Transactional(readOnly=true)` + 쓰기만 예외 | Step 6 |

## 3. 핵심 개념

### (1) JPA 변경 감지(dirty checking) — `save()` 없이 수정된다?

```java
@Transactional
public void updateArticle(Long id, ...) {
    Board board = boardRepository.findById(id).orElseThrow(...); // ① 영속 상태로 조회
    board.update(newTitle, newContent, filePath);               // ② 필드만 바꿈
    // ③ save() 안 해도 트랜잭션 끝에 UPDATE가 자동으로 나감!
}
```
- 트랜잭션 안에서 **조회한 엔티티(영속 상태)** 의 필드를 바꾸면, JPA가 "바뀐 걸" 감지해 **UPDATE SQL을 자동 실행**해요.
- 그래서 수정에는 `boardRepository.save(...)` 를 **부르지 않아도** 돼요. 이게 **변경 감지**예요.
- 반대로 새 글(작성)은 아직 영속 상태가 아니라 `save()`가 필요했죠. 차이를 기억해요.

### (2) 왜 setter 대신 `update()` 메서드?
| | `@Setter` 열기 | `update(...)` 메서드 |
|---|---|---|
| 변경 지점 | 아무 데서나 마구 바뀜(추적 어려움) | **"수정한다"는 의도**가 한 메서드에 모임 |
| 안전성 | `id`, `created`도 바뀔 위험 | 바꿀 것만(제목·내용·파일) 콕 집음 |

> 도메인 주도 스타일: 엔티티는 `@Getter`만 열고, 변경은 **뜻이 담긴 메서드**로 해요.

### (3) `fileFlag` — 파일 3경우를 구분하는 신호
수정 화면에서 첨부파일은 **세 가지** 경우가 있어요. 그런데 `file == null` 하나만으로는 "유지"와 "제거"를 구분할 수 없어요.

| 경우 | 사용자 행동 | `fileFlag` | `file` | 서버 처리 |
|------|------|------|------|------|
| 유지 | 파일 안 건드림 | `false` | 없음 | 기존 경로 그대로 |
| 교체 | 새 파일 올림 | `true` | 있음 | 기존 삭제 + 새로 저장 |
| 제거 | 첨부 지움 | `true` | 없음 | 기존 삭제 (첨부 없음) |

> 그래서 화면 JS가 파일을 건드리면 `fileFlag = true`를 함께 보내요. 서버는 이 깃발을 보고 분기해요.

### (4) 삭제 순서 — DB 먼저, 파일 나중
- 파일 삭제는 **되돌릴 수 없어요**(트랜잭션 롤백 대상이 아님).
- 그래서 **DB를 먼저 지우고, 파일을 나중에** 지워요. 만약 파일 삭제 중 예외가 나면 `@Transactional`이 DB 삭제까지 **롤백**해줘서 "글은 없는데 파일만 남는" 엇갈림을 막아요.

### (5) 단일 책임 원칙(SRP) — `FileService` 분리
```
[리팩터링 전] BoardService  = 게시글 로직 + 파일 저장/삭제/다운로드 (뒤죽박죽)
[리팩터링 후] BoardService  = 게시글 로직만
              FileService   = 파일 I/O만  ← BoardService가 "위임"해서 사용
```
> 게시글 로직과 파일 I/O는 **바뀌는 이유가 다르니** 나눠요. BoardService는 짧아지고, 파일 로직은 재사용하기 좋아져요.

## Step 1. 게시글 삭제 만들기 (DB + 파일, 순서 주의)

글을 지우면 **DB 행**과 **첨부파일**을 함께 지워요. 삭제할 파일 경로는 프론트가 JSON 본문으로 보내줘요.

- `dto/BoardDeleteRequestDto` : `filePath` 하나. **`@RequestBody`(JSON)** 로 받으니 setter가 필요해요.
- `service/BoardService.deleteArticle(id, request)` : `existsById`로 없으면 404, 있으면 `deleteById(id)` **후** 파일 삭제
- `controller/BoardApiController` : `DELETE /api/boards/{id}` + `@RequestBody`

<details>
<summary>힌트 보기</summary>

```java
// dto/BoardDeleteRequestDto.java
// @RequestBody는 Jackson이 setter로 값을 채운다. @Getter만 있으면 filePath가 null이 된다!
@Getter @Setter @NoArgsConstructor
public class BoardDeleteRequestDto {
    private String filePath;
}
```
```java
// service/BoardService.java  (삭제 부분)
@Transactional
public void deleteArticle(Long id, BoardDeleteRequestDto request) {
    if (!boardRepository.existsById(id)) {
        throw new BoardNotFoundException("게시글을 찾을 수 없습니다. id=" + id);
    }
    boardRepository.deleteById(id);        // ① DB 먼저
    deleteFile(request.getFilePath());     // ② 파일 나중 (Step 5에서 FileService로 옮김)
}

// 파일 삭제 (첨부 없던 글이면 건너뜀)
private void deleteFile(String filePath) {
    if (filePath == null || filePath.isBlank()) return;
    File file = new File(filePath);
    if (file.exists()) file.delete();
}
```
```java
// controller/BoardApiController.java  (삭제 부분)
@DeleteMapping("/{id}")
public void deleteArticle(@PathVariable long id, @RequestBody BoardDeleteRequestDto request) {
    boardService.deleteArticle(id, request);
}
```
> `BoardNotFoundException`은 3편(상세 조회)에서 만든 걸 재사용해요. `GlobalExceptionHandler`가 404로 바꿔줘요.
</details>

## Step 2. `Board.update()` 메서드 + 변경 감지 이해하기

수정에 쓸 **엔티티 메서드**를 만들어요. `Board`에는 `@Setter`가 없으니, 뜻이 담긴 `update()`로 값을 바꿔요.

- `domain/entity/Board` : `update(title, content, filePath)` 추가
- 작성자(`userId`)와 작성일(`created`)은 **건드리지 않아요.**
- 이 메서드로 필드만 바꾸면, 트랜잭션 끝에 JPA가 **UPDATE를 자동**으로 날려요(변경 감지). `save()` 불필요!

<details>
<summary>힌트 보기</summary>

```java
// domain/entity/Board.java  (메서드 추가)
public void update(String title, String content, String filePath) {
    this.title = title;
    this.content = content;
    this.filePath = filePath;
    // userId, created 는 수정 대상이 아니므로 그대로 둔다
}
```
> 핵심: 이 객체는 트랜잭션 안에서 `findById`로 가져온 **영속 상태**예요. 필드만 바꿔도 커밋 시 UPDATE가 나가요.
> (작성/신규 저장은 `save()`가 필요했지만, 수정은 변경 감지 덕에 `save()`가 필요 없어요.)
</details>

## Step 3. 게시글 수정 서비스 — `fileFlag`로 파일 3경우 처리

수정의 **핵심 로직**이에요. 제목·내용을 바꾸고, 파일은 `fileFlag`에 따라 유지/교체/제거해요.

- `dto/BoardUpdateRequestDto` : `title`, `content`, `boolean fileFlag`, `MultipartFile file` (**multipart라 `@ModelAttribute`** → setter 필요)
- `service/BoardService.updateArticle(id, request)` :
  - `findById`로 조회(없으면 404)
  - `filePath = board.getFilePath()` 를 기본값(유지)으로 두고,
  - `fileFlag`가 `true`면 기존 파일 삭제 후 `storeFile`로 새 경로(없으면 `null`=제거)
  - `board.update(...)` 호출

<details>
<summary>힌트 보기</summary>

```java
// dto/BoardUpdateRequestDto.java  (multipart 폼이라 @ModelAttribute로 받음 → setter 필요)
@Getter @Setter @NoArgsConstructor
public class BoardUpdateRequestDto {
    private String title;
    private String content;
    private boolean fileFlag;    // 파일을 건드렸는지(교체/제거) 여부
    private MultipartFile file;  // 새로 올린 파일 (교체할 때만 값이 있음)
}
```
```java
// service/BoardService.java  (수정 부분)
@Transactional
public void updateArticle(Long id, BoardUpdateRequestDto request) {
    Board board = boardRepository.findById(id)
            .orElseThrow(() -> new BoardNotFoundException("게시글을 찾을 수 없습니다. id=" + id));

    String filePath = board.getFilePath();   // 기본값: 기존 파일 유지
    if (request.isFileFlag()) {              // 파일을 건드렸다면
        deleteFile(board.getFilePath());     // 기존 파일 삭제
        filePath = storeFile(request.getFile()); // 새 파일 저장 (없으면 null = 제거)
    }
    board.update(request.getTitle(), request.getContent(), filePath); // 변경 감지로 UPDATE
}
```
> 기존 파일 경로는 **클라이언트가 보낸 값이 아니라 DB의 `board.getFilePath()`** 를 믿어요(위변조 방지).
> `boolean fileFlag`의 getter는 `isFileFlag()` 예요(롬복 규칙).
</details>

## Step 4. 수정 컨트롤러 — `PUT /api/boards/{id}`

프론트가 부르는 수정 API를 만들어요. 수정 대상 글의 `id`는 **본문이 아니라 URL 경로**로 받아요.

- `controller/BoardApiController` : `@PutMapping("/{id}")`
- `@PathVariable long id` + `@ModelAttribute BoardUpdateRequestDto request`

<details>
<summary>힌트 보기</summary>

```java
// controller/BoardApiController.java  (수정 부분)
@PutMapping("/{id}")
public void updateArticle(@PathVariable long id, @ModelAttribute BoardUpdateRequestDto request) {
    boardService.updateArticle(id, request);
}
```
> `PUT`은 "이 리소스를 이 내용으로 바꿔라"는 수정용 메서드예요. "무엇을" 수정할지(id)는 **경로**로 식별해요(RESTful).
> 파일이 섞인 폼이라 `@RequestBody`(JSON)가 아니라 **`@ModelAttribute`** 를 써요.
</details>

## Step 5. 리팩터링 — 파일 로직을 `FileService`로 분리

지금 `BoardService`에는 게시글 로직과 파일 로직(`storeFile`/`deleteFile`/`downloadFile`)이 뒤섞여 있어요. 파일 로직을 **`FileService`로 옮기고**, `BoardService`는 그걸 **위임**해서 쓰게 바꿔요.

- `service/FileService` : `@Value("${file.upload-dir}")` 와 `storeFile`/`deleteFile`/`downloadFile`을 이곳으로 이동(모두 `public`)
- `service/BoardService` : `private final FileService fileService;` 주입 → `fileService.storeFile(...)`, `fileService.deleteFile(...)` 로 호출
- 파일 관련 `import`(File, UUID 등)도 `BoardService`에서 정리

<details>
<summary>힌트 보기</summary>

```java
// service/FileService.java  (파일 I/O만 담당)
@Service
public class FileService {

    @Value("${file.upload-dir}")
    private String uploadDir;

    public String storeFile(MultipartFile file) { /* 기존 로직 그대로 이동 (public) */ }
    public void   deleteFile(String filePath)   { /* 기존 로직 그대로 이동 (public) */ }
    public Resource downloadFile(String fileName){ /* 기존 로직 그대로 이동 (public) */ }
}
```
```java
// service/BoardService.java  (파일은 FileService에 위임)
@Service
@RequiredArgsConstructor
public class BoardService {

    private final BoardRepository boardRepository;
    private final FileService fileService;   // ← 주입해서 위임

    // saveArticle:  String filePath = fileService.storeFile(file);
    // deleteArticle: fileService.deleteFile(request.getFilePath());
    // updateArticle: fileService.deleteFile(...) / fileService.storeFile(...)
}
```
> 왜 나눠요? 게시글 로직과 파일 I/O는 **바뀌는 이유가 달라요**(단일 책임 원칙). BoardService는 짧아지고, 파일 로직은 다른 기능에서도 재사용할 수 있어요.
> 다운로드 컨트롤러도 `fileService.downloadFile(...)`을 직접 쓰도록 바꿔주면 깔끔해요.
</details>

## Step 6. `@Transactional` 정리 + 통합 실행

마지막으로 트랜잭션 설정을 정돈하고, 화면에서 끝까지 확인해요.

- `BoardService` 클래스에 **`@Transactional(readOnly = true)`** 를 붙여 기본을 읽기 전용으로.
- 조회 메서드들의 메서드 레벨 어노테이션은 제거(클래스에서 상속).
- **쓰기 메서드(`saveArticle`/`updateArticle`/`deleteArticle`)에만** `@Transactional`을 붙여 덮어써요.

<details>
<summary>힌트 보기</summary>

```java
// service/BoardService.java
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)   // 클래스 기본 = 읽기 전용
public class BoardService {

    public List<Board> getBoardList(...) { ... }   // 조회 → 클래스 설정 상속(읽기 전용)
    public Board getBoardDetail(...)     { ... }   // 조회 → 상속

    @Transactional                                 // 쓰기 → 덮어쓰기(readOnly=false)
    public void saveArticle(...)   { ... }
    @Transactional
    public void updateArticle(...) { ... }
    @Transactional
    public void deleteArticle(...) { ... }
}
```
> `readOnly = true`면 Hibernate가 변경 감지용 스냅샷을 안 만들어 **가볍고**, 실수로 쓰기가 일어나는 것도 막아요.
> **메서드 레벨 설정이 클래스 레벨을 덮어써요**(더 가까운 쪽이 이김). `@Transactional`은 기본이 `readOnly=false`라 쓰기 가능해져요.

**통합 실행 체크**
1. 상세 → **수정** → 내용 바꾸고 저장 → 목록에서 바뀐 내용 확인 (콘솔에 UPDATE SQL 확인)
2. 파일 **교체** 후 저장 → `uploads` 폴더에서 기존 파일이 사라지고 새 파일이 생겼는지 확인
3. 상세 → **삭제** → 목록에서 사라지고, `uploads`의 첨부파일도 사라졌는지 확인
</details>
