# 로그인 만들기 (세션 + Optional)

> 1편에서 저장한 회원으로 **로그인**을 만들어요. 아이디/비밀번호가 맞으면 **서버가 "누가 로그인했는지"를 세션에 기억**하고, 게시판 목록으로 들어가요. 이 과정에서 값이 있을 수도/없을 수도 있는 결과를 다루는 **`Optional`** 과, 로그인 상태를 유지하는 **세션**을 익혀요.
> **아래 Step을 순서대로 따라가면, 화면에서 로그인 버튼을 눌러 세션이 만들어지고 목록 페이지로 이동하는 로그인이 완성됩니다.**
>
> 각 Step의 **힌트는 접혀 있어요.** "성공/실패를 어떻게 표현하지?", "로그인 상태는 어디에 기억하지?"를 먼저 떠올리고, 막히면 "힌트 보기"를 펼치세요.
>
> 1편과 마찬가지로 **시큐리티 미학습 단계**라, 비밀번호를 **평문으로 비교**해요. (실무에서는 암호화된 값끼리 비교해요. 뒤에서 배워요.)

---

## 0. 먼저 알아둘 점

- 이 과제는 **1편(회원가입)이 끝났다는 전제**예요. `Member` 엔티티 · `MemberRepository` · `MemberService` · `MemberApiController`가 이미 있어요.
- 아래는 제공돼요:
  - **프론트 화면**(`sign-in.html`)과 **자바스크립트**(`signIn.js`) — 화면은 건드리지 않아요
  - 로그인 폼을 보여주는 **뷰 컨트롤러**(`GET /members/login → sign-in`)
  - 로그인 성공 후 이동할 **목록 페이지 컨트롤러**(`GET "/"` → `board-list`) — 세션 값을 화면에 넘겨줘요
- **여러분이 작성할 것**: 리포지토리에 조회 메서드 1개 추가 · 로그인 DTO 2개 · 세션 키 상수 · 서비스 `login` · 컨트롤러 `login`.

> 프론트(`signIn.js`)가 이미 정해둔 **약속(계약)** 이에요. 여러분의 백엔드는 여기에 맞춰야 해요:
> - 요청: `POST /api/members/login` 으로 **폼 데이터**(`application/x-www-form-urlencoded`) `username`, `password` 를 보냄 (JSON 아님!)
> - 응답: `{ "loggedIn": true/false, "url": "...", "message": "..." }`
>   - `loggedIn: true` → JS가 `url`(`"/"`)로 이동
>   - `loggedIn: false` → JS가 `message`("아이디 또는 비밀번호가...")를 화면에 표시하고 이동 안 함

## 1. 무엇을 만드나요?

로그인 폼에 아이디/비밀번호를 넣고 **로그인**을 누르면 → 맞으면 **세션에 로그인 정보를 저장**하고 목록(`/`)으로 이동, 틀리면 **안내 메시지**를 띄우고 그대로 머물러요.

| 주소(메서드) | 하는 일 | 계층 |
|------|------|------|
| `GET /members/login` | 로그인 폼 보여주기 | 뷰 컨트롤러 (**제공**) |
| `POST /api/members/login` | 로그인 처리(검증 → 세션 저장) | **여러분이 만들 API** |
| `GET /` | 로그인한 사람만 보는 목록 | 뷰 컨트롤러 (**제공**) |

**완성되면 이렇게 동작해요**
```
[로그인 성공]
  아이디: hong / 비번: 1234  → [로그인]
  → 서버 세션에 userId=hong, userName=홍길동 저장 → 목록(/)으로 이동
  → 목록 상단: "홍길동님 환영합니다"

[로그인 실패]
  아이디: hong / 비번: 9999(틀림)  → [로그인]
  → 화면에 "아이디 또는 비밀번호가 일치하지 않습니다." (이동 안 함)
```

## 3. 핵심 개념

### (1) 세션(Session)이란? — "서버가 기억하는 로그인 상태"

HTTP는 원래 **요청이 끝나면 서로를 잊어버려요**(무상태). 그럼 "방금 로그인한 사람"을 어떻게 계속 알아볼까요? → **세션**을 써요.

```
로그인 성공 → 서버가 "세션 상자"를 하나 만들고 그 안에 userId를 넣음
           → 서버는 그 상자에 JSESSIONID(식별표) 를 붙여 브라우저에 줌
그 다음 요청 → 브라우저가 JSESSIONID 를 자동으로 같이 보냄
           → 서버는 그 번호로 "아, 아까 그 상자(=홍길동)" 를 찾아냄
```

| | 저장 위치 | 브라우저가 갖는 것 | 용도 |
|---|---|---|---|
| 세션 | **서버** 메모리 | `JSESSIONID`(식별표)뿐 | 로그인 상태 등 민감·일시 정보 |

> 즉 **실제 값(userId)은 서버에** 있고, 브라우저는 "내 상자 번호표"만 들고 다녀요. 그래서 값이 밖으로 새지 않아 로그인에 적합해요.

### (2) Optional — "값이 있을 수도, 없을 수도"

로그인 조회 결과는 **회원이 있을 수도(성공) 없을 수도(실패)** 있어요. 이걸 `null` 대신 `Optional`로 감싸 다뤄요.

| 상태 | 의미 |
|---|---|
| `Optional.of(회원)` | 값이 있음 |
| `Optional.empty()` | 값이 없음(빈 상자) |

자주 쓰는 메서드:
- `filter(조건)` : 값이 있고 조건도 참이면 유지, 아니면 **비운다**(empty)
- `map(변환)` : 값이 있으면 다른 값으로 바꾼다, 없으면 그대로 empty
- `orElseGet(함수)` : 값이 없을 때만 함수를 실행해 대체값을 만든다

### (3) orElse vs orElseGet — 실행 "시점"이 다르다!
```java
.orElse(LoginResponseDto.fail())      // fail()을 "항상 미리" 실행해 둠 (값이 있어도 낭비 실행)
.orElseGet(LoginResponseDto::fail)    // 값이 "없을 때만" 그제서야 fail() 실행 (효율적)
```
> 로그인 성공 시엔 실패 객체를 만들 필요가 없죠? 그래서 **`orElseGet`** 이 맞아요.
> `LoginResponseDto::fail` 은 "지금 실행"이 아니라 "**이 메서드를 나중에 쓰라**"고 넘기는 **메서드 참조**예요(괄호 없음에 주목).

### (4) 정적 팩토리 메서드 — 생성자 대신 "이름 있는 생성기"
```java
LoginResponseDto.success();   // "성공용 응답" — 이름만 봐도 의도가 드러남
LoginResponseDto.fail();      // "실패용 응답"
```
> `new LoginResponseDto(true, "/", "...")` 처럼 인자를 나열하는 대신, **의미가 담긴 이름**으로 만들면 읽기 쉬워요.

## Step 1. 아이디로 회원 조회하기 (findByUserId)

로그인하려면 "이 아이디의 회원"을 DB에서 찾아야 해요. `MemberRepository`에 조회 메서드를 **이름 규칙**으로 추가해요.

- `MemberRepository`에 `findByUserId(String)` 추가 → 반환은 `Optional<Member>`
- 없을 수도 있으니 `Member`가 아니라 `Optional<Member>`로 감싸 받아요.

<details>
<summary>힌트 보기</summary>

```java
// domain/repository/MemberRepository.java
public interface MemberRepository extends JpaRepository<Member, Long> {

    boolean existsByUserId(String userId);              // 1편에서 만든 것

    // [find + By + UserId] → user_id 로 1건 조회. 없을 수 있으므로 Optional 로 감싼다
    Optional<Member> findByUserId(String userId);       // SELECT * FROM member WHERE user_id = ?
}
```
> `find`(조회) + `By` + `UserId`(조건 필드). 필드 이름은 엔티티의 `userId`와 **정확히** 같아야 해요.
</details>

## Step 2. 로그인 요청/응답 DTO 만들기

로그인 폼 값을 담을 **요청 DTO**와, 성공/실패 결과를 담을 **응답 DTO**를 만들어요. 응답은 **정적 팩토리 메서드**로 성공/실패를 깔끔하게 구분해요.

- `dto/LoginRequestDto` : `username, password` (폼 바인딩용 `@Setter`/기본생성자)
- `dto/LoginResponseDto` : `loggedIn, url, message` + `success()` / `fail()` 정적 팩토리, 생성자는 `private`

<details>
<summary>힌트 보기</summary>

```java
// dto/LoginRequestDto.java  — 폼(x-www-form-urlencoded)을 @ModelAttribute 로 받으므로 setter+기본생성자 필요
@Getter @Setter @NoArgsConstructor
public class LoginRequestDto {
    private String username;   // 폼의 name="username"
    private String password;   // 폼의 name="password"
}
```
```java
// dto/LoginResponseDto.java
@Getter
@AllArgsConstructor(access = AccessLevel.PRIVATE)   // 생성자를 막고, 아래 정적 팩토리로만 만들게 한다
public class LoginResponseDto {
    private boolean loggedIn;
    private String url;
    private String message;

    public static LoginResponseDto success() {
        return new LoginResponseDto(true, "/", "로그인에 성공했습니다.");
    }
    public static LoginResponseDto fail() {
        return new LoginResponseDto(false, null, "아이디 또는 비밀번호가 일치하지 않습니다.");
    }
}
```
> 성공은 `url="/"`(목록으로 이동), 실패는 `url=null`(이동 안 함). 프론트가 `loggedIn`을 보고 분기해요.
</details>

## Step 3. 세션 키를 상수로 모으기 (SessionConst)

세션에 값을 넣을 때 `"userId"` 같은 **문자열 키**를 써요. 그런데 저장하는 쪽과 꺼내는 쪽이 **철자가 한 글자라도 다르면** 값이 `null`이 되는 **조용한 버그**가 생겨요. 그래서 키를 **상수 한곳**에 모아요.

- `constant/SessionConst` : `USER_ID`, `USER_NAME` 상수 + `private` 생성자(인스턴스화 방지)

<details>
<summary>힌트 보기</summary>

```java
// constant/SessionConst.java
public class SessionConst {

    private SessionConst() {}   // new 로 만들 필요 없는 상수 모음이라 생성자를 막는다

    public static final String USER_ID = "userId";
    public static final String USER_NAME = "userName";
}
```
> 저장할 때도, 꺼낼 때도 **이 상수를 그대로** 써요. 그럼 오타로 키가 어긋날 일이 없어요.
> (로그인 컨트롤러는 저장하고, 제공된 목록 컨트롤러는 이 상수로 꺼내 화면에 넘겨요. 양쪽이 **같은 상수**를 봐야 해요.)
</details>

## Step 4. 서비스 — 로그인 판정 로직 만들기

`MemberService.login()`은 아이디로 회원을 찾고, **비밀번호가 일치하면** 그 회원을, 아니면 **빈 결과**를 돌려줘요. 성공/실패를 예외가 아니라 **`Optional`의 값 유무**로 표현하는 게 포인트예요.

- `login(LoginRequestDto)` : 반환 `Optional<Member>`
- `findByUserId(...)` 결과에 `.filter(...)`로 비밀번호 일치까지 확인
- 조회만 하므로 `@Transactional(readOnly = true)`

<details>
<summary>힌트 보기</summary>

```java
// service/MemberService.java (login 메서드 추가)
@Transactional(readOnly = true)
public Optional<Member> login(LoginRequestDto request) {
    return memberRepository.findByUserId(request.getUsername())
            .filter(member -> member.getPassword().equals(request.getPassword()));
}
```
- `findByUserId` 가 회원을 찾으면 → `Optional`에 회원이 담김
- `.filter(비밀번호 일치?)` 가 참이면 그대로 유지, 틀리면 **비운다(empty)**
- 결국: **아이디도 있고 비번도 맞아야** 값이 남아요 = 로그인 성공

> `member.getPassword().equals(...)` 는 **평문 비교**예요(시큐리티 미학습). 실무에선 암호화 값끼리 비교해요.
> 람다를 안 쓰면 `if (member != null && member.getPassword().equals(...))` 로 풀어 쓴 것과 같아요.
</details>

## Step 5. 컨트롤러 — 세션 저장 + 성공/실패 응답

프론트가 부르는 `POST /api/members/login`을 만들어요. 서비스 결과(`Optional`)가 **있으면 세션에 저장 후 성공**, **없으면 실패**를 응답해요.

- `MemberApiController.login` : `@PostMapping("/login")`, `@ModelAttribute LoginRequestDto` + `HttpSession`
- `.map(...)` : 값이 있을 때만 실행 → 세션에 `USER_ID`, `USER_NAME` 저장 후 `success()`
- `.orElseGet(LoginResponseDto::fail)` : 값이 없을 때만 실행 → `fail()`

<details>
<summary>힌트 보기</summary>

```java
// controller/MemberApiController.java (login 메서드 추가)
@PostMapping("/login")
public LoginResponseDto login(@ModelAttribute LoginRequestDto request, HttpSession session) {
    return memberService.login(request)
            .map(member -> {
                // 로그인 성공 → 세션에 저장 (SessionConst 상수 사용!)
                session.setAttribute(SessionConst.USER_ID, member.getUserId());
                session.setAttribute(SessionConst.USER_NAME, member.getUserName());
                return LoginResponseDto.success();
            })
            .orElseGet(LoginResponseDto::fail);   // 실패 → 이동 없이 메시지만
}
```
- `@ModelAttribute` : 폼 데이터(JSON 아님!)를 DTO에 담아요. `@RequestBody`를 쓰면 안 돼요.
- `HttpSession` : 파라미터로 선언하면 스프링이 세션을 넣어줘요. `setAttribute(키, 값)`으로 저장.
- `.map`은 성공 때만, `.orElseGet`은 실패 때만 → **if/else 없이** 분기돼요.

> 왜 `orElse`가 아니라 `orElseGet`? → 성공했을 땐 `fail()`을 만들 필요가 없으니까요(3-(3) 참고).
</details>

## Step 6. 화면에서 끝까지 확인하기 (통합 실행)

로그인 → 목록 진입까지 실제 화면으로 돌려봐요.

1. 앱 실행 후 `GET /members/login` 접속 (로그인 폼).
2. 1편에서 가입한 아이디/비번으로 로그인 → **목록(`/`)으로 이동**하고 상단에 **이름 환영 문구**가 보이면 성공.
3. 비밀번호를 **틀리게** 입력 → **안내 메시지**가 뜨고 이동하지 않으면 성공.
4. (보너스) 로그인 안 한 채 `/` 로 바로 들어가 보면, 제공된 목록 화면이 세션이 없어 로그인 페이지로 되돌려 보내요(세션 기반 접근 제어).

<details>
<summary>잘 안 될 때 체크리스트</summary>

- 로그인이 늘 실패함: 서비스에서 `@ModelAttribute`로 받은 `username/password`가 폼 name과 같은지, `.filter` 비교가 맞는지
- 400/415: 로그인은 **폼 데이터**라 `@RequestBody`가 아니라 **`@ModelAttribute`** 여야 해요
- 이동은 되는데 이름이 안 뜸: 세션에 `USER_NAME`을 저장했는지, 목록 화면이 꺼내는 키(`SessionConst.USER_NAME`)와 **같은 상수**인지
- 이름이 null: 저장 쪽과 꺼내는 쪽 키 철자 불일치 → 반드시 `SessionConst` 상수를 양쪽에서 사용
- 실패 메시지가 안 뜸: 응답이 `{loggedIn:false, message:...}` 형태인지 (프론트가 `loggedIn`으로 분기)
</details>

## 완성 체크리스트
- [ ] `MemberRepository.findByUserId` (`Optional<Member>`)
- [ ] `LoginRequestDto`(폼 바인딩) + `LoginResponseDto`(정적 팩토리 `success`/`fail`)
- [ ] `SessionConst`(USER_ID, USER_NAME 상수)
- [ ] `MemberService.login` (`Optional.filter`로 비밀번호 판정, `readOnly`)
- [ ] `MemberApiController.login` (`@ModelAttribute` + `HttpSession`, `map`/`orElseGet`)
- [ ] 화면에서 로그인 성공(목록 이동+환영) / 실패(메시지) 둘 다 확인
