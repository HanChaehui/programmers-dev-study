# 게시판 읽기 만들기 (목록 페이징 + 상세 조회)

> 이제 **게시판**을 시작해요. 첫걸음은 "읽기(Read)"예요. 글 목록을 **페이지 단위**로 끊어서 보여주고, 글 하나를 눌러 **상세**로 들어가는 흐름을 만들어요.
> **아래 Step을 순서대로 따라가면, 목록 페이지에서 글들이 페이지별로 보이고 글을 눌러 상세를 여는 화면이 완성됩니다.**
>
> 각 Step의 **힌트는 접혀 있어요.** "이 조회는 어떤 반환 타입이어야 하지?"를 먼저 떠올리고, 막히면 "힌트 보기"를 펼치세요.
>
> 이번 편은 **읽기(조회)만** 다뤄요. 글쓰기·파일 업로드는 다음 편(4편)에서 해요. 그래서 `Board` 엔티티에 `filePath` 필드는 만들되, 지금은 항상 비어(NULL) 있어요.

---

## 0. 먼저 알아둘 점

- 이 과제는 **1~2편(회원가입·로그인)을 끝냈다는 전제**예요. 계층 구조(컨트롤러→서비스→리포지토리), JPA, DTO, `@RestControllerAdvice`는 이미 익혔다고 봐요.
- 아래는 제공돼요:
  - **프론트 화면**: `board-list.html`(+`boardList.js`), `board-detail.html`(+`boardDetail.js`) — 화면은 건드리지 않아요
  - **뷰 컨트롤러**: 목록 화면 `GET /`, 상세 화면 `GET /detail` (모델에 `id` 전달)
  - **시드 데이터**: `board` 테이블에 미리 넣어둔 글 20개 (`data.sql`)
- **여러분이 작성할 것**: `Board` 엔티티 · `BoardRepository` · 목록/상세 **서비스** · 응답 **DTO** · **API 컨트롤러** · 없는 글 조회 시 **404 예외 처리**.

> 프론트가 이미 정해둔 **약속(계약)** 이에요. 여러분의 API는 여기에 맞춰야 해요:
> - 목록: `GET /api/boards?page=1&size=10` → `{ "boards": [...], "last": false, "totalPages": 2 }`
> - 상세: `GET /api/boards/{id}` → `{ "title": "...", "content": "...", "created": "...", "userId": "...", "filePath": null }`
> - 없는 글: 상태코드 **404** + `{ "message": "..." }`

## 1. 무엇을 만드나요?

목록 페이지에 글이 **최신순으로 10개씩** 보이고, 하단 페이지 번호로 넘겨봐요. 글 제목을 누르면 **상세 페이지**에서 제목·내용·작성자·작성일을 봐요. 없는 글 주소로 가면 안내를 받아요.

| 주소(메서드) | 하는 일 | 계층 |
|------|------|------|
| `GET /` | 목록 화면 보여주기 | 뷰 컨트롤러 (**제공**) |
| `GET /detail?id=..` | 상세 화면 보여주기 | 뷰 컨트롤러 (**제공**) |
| `GET /api/boards` | 목록 데이터(페이징) 주기 | **여러분이 만들 API** |
| `GET /api/boards/{id}` | 글 1건 데이터 주기 | **여러분이 만들 API** |

**완성되면 이렇게 동작해요**
```
[목록]  /  접속
  20번째 글 ────────────  (최신이 위)
  19번째 글 ────────────
  ... (10개) ...
  11번째 글 ────────────
  [1] [2]     ← 페이지 번호

[상세]  글 제목 클릭 → /detail?id=20
  제목: 20번째 글
  작성자: hong | 작성일: 2026-06-24 08:43
  내용: ......

[없는 글]  /api/boards/999
  → 404 + "게시글을 찾을 수 없습니다. id=999"
```

## 2. 학습 목표

| 개념 | 어디서 배우나 |
|------|------|
| `@Entity`에 다양한 컬럼 매핑(TEXT, nullable, 날짜) | Step 1 (`Board`) |
| 날짜를 사람이 읽기 좋게: `@JsonFormat` | Step 1 (`created`) |
| `JpaRepository`가 주는 **페이징**(`findAll(Pageable)`) | Step 2~3 |
| 화면 페이지(1부터) ↔ `Pageable`(0부터) 변환 | Step 3 (`getBoardList`) |
| 목록 응답 DTO에 페이지 정보(`last`, `totalPages`) 담기 | Step 4 (`BoardApiController`) |
| `findById`의 `Optional` + `orElseThrow`로 404 | Step 5 (`getBoardDetail`) |
| 기존 `@RestControllerAdvice`에 404 핸들러 추가 | Step 5 (`GlobalExceptionHandler`) |

## 3. 핵심 개념

### (1) 페이징 — "몇 페이지를, 몇 개씩, 어떤 순서로"

글이 1000개면 한 번에 다 내려줄 수 없어요. **잘라서(페이지 단위)** 조회해요. 스프링 데이터 JPA는 이걸 `Pageable`로 표현해요.

```java
Pageable pageable = PageRequest.of(0, 10, Sort.by("id").descending());
//                                 ↑    ↑         ↑
//                              0페이지 10개씩   id 내림차순(최신 먼저)
```

> **가장 헷갈리는 함정**: 화면의 페이지 번호는 보통 **1부터** 시작하지만, `Pageable`은 **0부터** 시작해요. 그래서 서비스에서 `page - 1`로 바꿔줘야 해요. (화면 1페이지 = `Pageable` 0페이지)

`findAll(pageable)`의 반환 타입은 `Page<Board>`예요. 목록만이 아니라 **페이지 정보**까지 담긴 "상자"예요.

| `Page<Board>`가 주는 것 | 뜻 |
|---|---|
| `getContent()` | **이번 페이지의 글 목록** (`List<Board>`) |
| `getTotalElements()` | 전체 글 개수 |
| `getTotalPages()` | 전체 페이지 수 |
| `isLast()` | 마지막 페이지 여부 |

> 헷갈림 주의: `page.getContent()`의 'content'는 **"페이지에 담긴 목록"**이고, `board.getContent()`의 'content'는 **"글의 본문"**이에요. 단어만 같지 완전히 다른 거예요!

### (2) `findById`는 왜 `Optional`을 줄까?
```java
Optional<Board> result = boardRepository.findById(id);
```
- 그 id의 글이 **있을 수도, 없을 수도** 있어요. 없으면 `null` 대신 **빈 Optional**을 줘서 NPE를 예방해요.
- 우리는 "없으면 예외를 던진다"로 처리해요 → `orElseThrow(...)`.

```java
return boardRepository.findById(id)
        .orElseThrow(() -> new BoardNotFoundException("게시글을 찾을 수 없습니다. id=" + id));
```
> 값이 있으면 그 `Board`를, 없으면 `BoardNotFoundException`을 던져요. 이 예외는 `@RestControllerAdvice`가 **404**로 바꿔줘요(1편에서 만든 그 방식 그대로).

### (3) `@JsonFormat` — 날짜를 사람 친화적으로
- `LocalDateTime`을 그냥 JSON으로 내리면 `2026-06-24T08:43:00`처럼 중간에 **`T`**가 붙어요(ISO-8601 형식).
- `@JsonFormat(pattern = "yyyy-MM-dd HH:mm")`을 붙이면 `2026-06-24 08:43`처럼 보기 좋게 나가요.

| | 지정 안 하면 | `@JsonFormat` 지정 |
|---|---|---|
| 출력 | `2026-06-24T08:43:00` | `2026-06-24 08:43` |

## Step 1. 게시글 엔티티 만들기

`board` 테이블과 짝이 되는 `Board` 클래스를 만들어요. 회원 엔티티보다 컬럼 종류가 다양해요(긴 본문 TEXT, 날짜, 널 허용 필드).

- `domain/entity/Board` : `id, title, content, userId, filePath, created`
- `content`는 길 수 있으니 `TEXT`, `filePath`는 첨부 없을 수 있으니 **nullable**
- `created`(작성일)에는 **`@JsonFormat`**을 붙여 사람이 읽기 좋은 형식으로

<details>
<summary>힌트 보기</summary>

```java
// domain/entity/Board.java
@Entity
@Table(name = "board")
@Getter
@Builder
@AllArgsConstructor
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Board {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 100)
    private String title;

    @Column(nullable = false, columnDefinition = "TEXT") // 본문은 길 수 있어 TEXT
    private String content;

    @Column(nullable = false, length = 30)
    private String userId;

    @Column(length = 255)
    private String filePath; // 첨부파일 경로 - 없을 수 있어 nullable

    // 안 붙이면 2026-06-24T08:43:00 처럼 'T'가 붙는다 → pattern으로 사람 친화적 표기
    @JsonFormat(pattern = "yyyy-MM-dd HH:mm")
    private LocalDateTime created;
}
```
> `filePath`는 이번 편에선 항상 NULL이에요(글쓰기가 아직 없으니까). 4편에서 채워요.
</details>

## Step 2. 게시글 리포지토리 만들기

`BoardRepository`는 특별한 메서드가 필요 없어요. `JpaRepository`를 상속하면 **페이징 조회**(`findAll(Pageable)`)와 **개수 세기**(`count()`)가 이미 딸려오거든요.

- `domain/repository/BoardRepository` : `JpaRepository<Board, Long>` 상속, **본문 비움**

<details>
<summary>힌트 보기</summary>

```java
// domain/repository/BoardRepository.java
public interface BoardRepository extends JpaRepository<Board, Long> {
    // 커스텀 메서드 없음!
    // findAll(Pageable), count(), findById() 모두 JpaRepository가 이미 제공한다
}
```
> `JpaRepository`는 `PagingAndSortingRepository`도 상속하고 있어서 `findAll(Pageable)`이 공짜로 들어와요.
</details>

## Step 3. 목록 서비스 — 페이지로 잘라 조회하기

`BoardService`에 목록 조회와 전체 개수 조회를 만들어요. **화면 page(1부터)를 `Pageable`(0부터)로 바꾸는 것**이 이번 Step의 핵심이에요.

- `getBoardList(page, size)` : `PageRequest.of(page - 1, size, Sort.by("id").descending())` → `findAll(pageable).getContent()`
- `getTotalBoards()` : `count()` (마지막 페이지 계산에 필요)
- 조회만 하므로 `@Transactional(readOnly = true)`

<details>
<summary>힌트 보기</summary>

```java
// service/BoardService.java
@Service
@RequiredArgsConstructor
public class BoardService {

    private final BoardRepository boardRepository;

    // 최신 글이 위로 오도록 id 내림차순. 화면 page(1~)를 Pageable(0~)로 바꾸려 page - 1
    @Transactional(readOnly = true)
    public List<Board> getBoardList(int page, int size) {
        Pageable pageable = PageRequest.of(page - 1, size, Sort.by("id").descending());
        return boardRepository.findAll(pageable).getContent(); // 이번 페이지의 글 목록
    }

    @Transactional(readOnly = true)
    public int getTotalBoards() {
        return (int) boardRepository.count();
    }
}
```
> `getContent()`의 content는 "페이지에 담긴 목록"이지, `Board.content`(본문)와 전혀 관계없어요!
</details>

## Step 4. 목록 컨트롤러 — 페이지 정보까지 담아 응답하기

`GET /api/boards`를 만들어요. 목록뿐 아니라 **마지막 페이지 여부(`last`)와 전체 페이지 수(`totalPages`)**를 계산해 함께 내려줘야 프론트가 페이지 번호를 그려요.

- `dto/BoardListResponseDto` : `boards, last, totalPages` (`@Builder`)
- `BoardApiController` : `@RestController`, `@RequestMapping("/api/boards")`
- `getBoardList` : `page`(기본 1), `size`(기본 10) 파라미터. `totalPages`는 **올림** 계산.

<details>
<summary>힌트 보기</summary>

```java
// dto/BoardListResponseDto.java
@Getter @Builder
public class BoardListResponseDto {
    private List<Board> boards;
    private boolean last;
    private int totalPages;
}
```
```java
// controller/BoardApiController.java
@RestController
@RequestMapping("/api/boards")
@RequiredArgsConstructor
public class BoardApiController {

    private final BoardService boardService;

    @GetMapping
    public BoardListResponseDto getBoardList(
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "10") int size
    ) {
        List<Board> boards = boardService.getBoardList(page, size);
        int totalBoards = boardService.getTotalBoards();

        // 전체 글 수 ÷ 페이지당 개수 → "올림" (21개/10 = 3페이지)
        int totalPages = (int) Math.ceil((double) totalBoards / size);
        boolean last = page >= totalPages;

        return BoardListResponseDto.builder()
                .boards(boards)
                .last(last)
                .totalPages(totalPages)
                .build();
    }
}
```
> 왜 `Math.ceil`? 정수끼리 나누면 소수점이 버려져요. 21/10=2가 아니라 3페이지가 필요하니 `double`로 나눈 뒤 올림해요.
</details>

## Step 5. 상세 조회 + 없는 글 404 처리

`GET /api/boards/{id}`를 만들어요. `findById`로 1건을 찾되, **없으면 404**로 응답해요.

- `service/BoardService.getBoardDetail(id)` : `findById(id).orElseThrow(() -> new BoardNotFoundException(...))`
- `exception/BoardNotFoundException` : `RuntimeException` 상속
- `GlobalExceptionHandler`에 **404 핸들러 추가** (1편에서 만든 클래스에 메서드만 더함)
- `dto/BoardDetailResponseDto` : `title, content, created(@JsonFormat), userId, filePath` (`@Builder`)
- 컨트롤러에 `@GetMapping("/{id}")` 추가 (`@PathVariable`)

<details>
<summary>힌트 보기</summary>

```java
// service/BoardService.java (메서드 추가)
@Transactional(readOnly = true)
public Board getBoardDetail(Long id) {
    return boardRepository.findById(id)
            .orElseThrow(() -> new BoardNotFoundException("게시글을 찾을 수 없습니다. id=" + id));
}
```
```java
// exception/BoardNotFoundException.java
public class BoardNotFoundException extends RuntimeException {
    public BoardNotFoundException(String message) { super(message); }
}
```
```java
// exception/GlobalExceptionHandler.java (핸들러 추가)
@ExceptionHandler(BoardNotFoundException.class)
public ResponseEntity<ErrorResponseDto> handleBoardNotFound(BoardNotFoundException e) {
    return ResponseEntity
            .status(HttpStatus.NOT_FOUND) // 404
            .body(new ErrorResponseDto(HttpStatus.NOT_FOUND.value(), e.getMessage()));
}
```
```java
// dto/BoardDetailResponseDto.java
@Getter @Builder
public class BoardDetailResponseDto {
    private String title;
    private String content;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm")
    private LocalDateTime created;

    private String userId;
    private String filePath;
}
```
```java
// controller/BoardApiController.java (메서드 추가)
@GetMapping("/{id}")
public BoardDetailResponseDto getBoardDetail(@PathVariable long id) {
    Board board = boardService.getBoardDetail(id);
    return BoardDetailResponseDto.builder()
            .title(board.getTitle())
            .content(board.getContent())
            .created(board.getCreated())
            .userId(board.getUserId())
            .filePath(board.getFilePath())
            .build();
}
```
> 404 상태코드는 "그 리소스(글)를 못 찾음"을 뜻해요. 1편의 409(중복)와 마찬가지로, 상황에 맞는 상태코드를 돌려주는 게 RESTful이에요.
</details>

## Step 6. 화면에서 끝까지 확인하기 (통합 실행)

이제 조각들을 합쳐 **실제 화면**으로 돌려봐요.

1. 앱을 실행하고 `GET /` 접속 → 시드 글 20개가 **최신순 10개씩** 보이고, 하단에 `[1] [2]` 페이지 번호가 뜨면 성공.
2. `[2]`를 눌러 다음 페이지로 넘어가는지 확인.
3. 글 제목을 클릭 → `/detail?id=..`에서 **제목·작성자·작성일·내용**이 보이면 성공. 날짜가 `2026-06-24 08:43`처럼 `T` 없이 나오는지 확인.
4. 주소창에 `/api/boards/999`(없는 글) 입력 → **404 + 메시지**가 나오면 성공.

<details>
<summary>잘 안 될 때 체크리스트</summary>

- 목록이 비어 보임: `data.sql` 시드가 실행됐는지(`SELECT COUNT(*) FROM board;`), `ddl-auto`/초기화 설정 확인
- 순서가 반대(오래된 글이 위): `Sort.by("id").descending()` 확인
- 1페이지인데 빈 화면 / 글이 안 맞음: 화면 page를 그대로 넘겼는지 → **`page - 1`** 로 변환했는지 확인
- 페이지 번호가 안 그려짐: 응답에 `totalPages`, `last`가 들어있는지 (프론트가 이 값으로 번호를 그려요)
- 날짜에 `T`가 보임: `created`에 `@JsonFormat(pattern = "yyyy-MM-dd HH:mm")`을 붙였는지
- 없는 글인데 500이 뜸: `GlobalExceptionHandler`에 `BoardNotFoundException` 핸들러(404)를 추가했는지
</details>
