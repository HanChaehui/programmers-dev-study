## 다형성 - 가계부 과제 페이지 참고

- 동일한 인터페이스나 부모 클래스를 공유하는 여러 객체들이 각자 다른 방식으로 동작하도록 할 수 있는 기능
- “하나의 인터페이스로 여러 형태의 객체를 처리할 수 있다”는 의미를 가짐
- 이를 통해 코드의 유연성과 확장성을 크게 향상시킬 수 있음

```java
				K_animal myAnimal = new K_animal(); // K_animal 타입의 객체
        K_animal myDog = new K_dog(); // Dog타입의 객체, Animal타입으로 업캐스팅

        myAnimal.sound();
        myDog.sound(); // 런타임 시점에 Dog의 sound()호출

        // 다운캐스팅을 통해 다시 Dog 타입으로 변환
				// K_dog myDog2 = (K_dog) myAnimal; // 실질적인 인스턴스는 animal -> dog(x)
        K_dog myDog2 = (K_dog) myDog; // 실질적인 인스턴스가 dog이기 때문에 다운캐스팅 가능
        myDog2.fetch();

```

왼쪽 타입 = 사용할 수 있는 메서드 범위 결정
오른쪽 객체 = 실제 실행되는 메서드 결정

## 추상화 Abstract

추상화는 객체지향 프로그래밍의 핵심 개념 중 하나로, 공통적인 속성과 기능을 뽑아내어 상위 개념으로 정의하는 것

자바에서는 추상 클래스와 추상 메서드를 통해 추상화를 구현할 수 있다.

#### 추상 클래스 `abstract class 클래스명 {…}`

- 인스턴스화할 수 없는 클래스
- 추상 클래스 자체로 객체를 생성할 수 없으며, 반드시 이 클래스를 상속받은 자식 클래스를 통해서만 객체를 생성할 수 있음
- 일반 메서드, 추상메서드, 필드(멤버 변수), 생성자를 모두 가질 수 있음
- 여러 자식 클래스에서 공통으로 사용할 기능을 미리 정의하고, 일부 기능은 자식 클래스에서 반드시 구현하도록 강제할 때 사용한다.

#### 추상 메서드 `abstract return 메서드명 (…);`

- 선언만 존재하고, 구현부가 없는 메서드
- 추상 클래스를 상속받은 자식 클래스는 반드시 이 메서드를 오버라이딩하여 구현해야 하지만 자식 클래스 또한 추상 클래스로 선언하다면 구현하지 않아도 됨
- 추상 메서드를 포함하는 클래스는 반드시 추상 클래스로 선언해야함

#### 추상 클래스의 장점

- 공통 기능을 부모 클래스에서 미리 구현할 수 있음
- 자식 클래스들이 반드시 구현해야 하는 기능을 강제할 수 있음
- 관련 있는 클래스들을 하나의 상위 타입으로 묶어 관리할 수 있음
- 코드 중복을 줄이고, 클래스 구조를 더 체계적으로 만들 수 있음

#### 추상 클래스의 단점

1. 단일 상속의 제한 
    - 자바는 단일 상속만 지원하므로, 한 클래스가 두 개 이상의 추상 클래스를 상속 받을 수 없음
    - 여러 부모 클래스의 기능을 동시에 상속받고 싶을 때 제약이 생김
    - 이 경우, 인터페이스를 사용해야 하지만, 인터페이스는 구현을 가질 수 없다는 다른 제약이 있음 (Java 8 이후의 인터페이스는 `default` 메서드를 통해 일부 구현을 가질 수 있음)
2. 구현의 강제성 
    - 추상 메서드를 상속받는 모든 자식 클래스는 해당 메서드를 반드시 구현해야 함
    - 하지만 모든 자식 클래스에 그 메서드가 꼭 필요한 것은 아닐 수 있음
    - 이 경우 불필요한 오버라이딩이 발생할 수 있음
    - 따라서 추상 메서드는 모든 자식 클래스에 공통적으로 필요한 기능일 때 선언하는 것이 좋음
3. 유연성 부족 
    - 추상 클래스는 주로 공통 기능을 제공하기 위해 사용되지만, 때로는 이 공통 기능이 모든 하위 클래스에 적합하지 않을 수 있음
    - 부모 클래스에 너무 많은 역할이 들어가면 자식 클래스들이 원하지 않는 기능까지 상속받게 됨
    - 이 경우 클래스 구조가 복잡해지고 유지보수가 어려워질 수 있음
    - 따라서 추상 클래스에는 자식 클래스들이 실제로 공유하는 공통 기능만 넣는 것이 좋음

#### 정리

- 추상 클래스는 공통 기능을 제공하면서, 일부 기능은 자식 클래스에서 반드시 구현하도록 강제하는 클래스이다.
- 추상 메서드는 구현 없이 선언만 존재하는 메서드이다.
- 추상 클래스는 객체를 직접 생성할 수 없으며, 상속을 통해 사용해야 한다.
- 공통 기능이 많고 클래스 간 관계가 명확할 때 추상 클래스를 사용하면 좋다.
- 여러 종류의 기능을 유연하게 조합해야 한다면 인터페이스 사용을 고려하는 것이 좋다.

## 인터페이스 Interface

- 클래스들이 구현해야 하는 메서드의 집합을 정의하는 데에 사용되는 추상 타입
- 메서드 선언만, 구현은 미포함
- 해당 인터페이스를 구현하는 클래스는 인터페이스에 선언된 모든 메서드를 반드시 구현해야함
- **상속과 달리 멤버변수와 생성자를 가질 수 없음**

## 인터페이스의 필요성

1. 표준화
    
    여러 클래스가 동시에 동일한 메서드를 구현하도록 강제할 수 있고, 이는 일관된 API 설계와 코드 표준화를 가능하게 함
    
2. 다중 상속의 대안
    
    자바는 클래스의 다중 상속을 지원하지 않지만, 인터페이스는 여러 개를 구현할 수 있음, 이를 통해 다중 상속의 이점을 제공하면서 다중 상속에서 발생할 수 있는 복잡성과 충돌을 피할 수 있음
    
3. 유연한 설계
    
    특정 클래스의 구현에 의존하지 않고, 인터페이스를 기반으로 프로그래밍할 수 있음, 이는 코드의 유연성과 확장성을 크게 향상시킴
    
4. 느슨한 결합
    
    시스템의 각 부분을 독립적으로 개발하고 변경할 수 있게 함
    

## ⭐인터페이스와 추상 클래스의 차이점

- 다중 구현 : 클래스는 여러 인터페이스를 구현할 수 있지만, 추상 클래스는 하나만 상속 가능
- 구현 유무 : 인터페이스는 구현을 포함하지 않지만(default는?), 추상 클래스는 일부 메서드의 구현을 포함할 수 있음
- 상태 유지 : 인터페이스는 생성자와 상태(멤버변수)를 가질 수 없지만, 추상 클래스는 가질 수 있음
- 인터페이스는 기능 명세서 느낌
- 상속(, 추상 클래스)과 마찬가지로 인터페이스 타입의 변수로 여러 구현체를 참조 가능
- 왼쪽 타입 = 사용할 수 있는 메서드 범위 결정
오른쪽 객체 = 실제 실행되는 메서드 결정

## List

- 순서가 있는 요소들의 컬렉션을 나타내는 인터페이스
- List 인터페이스를 구현하는 대표적인 클래스 : ArrayList, LinkedList, Vector 등
- List는 순서 구분, 중복 허용, 인덱스로 요소 접근 가능

#### List 주요 메서드

- add(E e) : 리스트에 요소 추가
- get(int index) : 인덱스에 있는 요소 반환
- remove(int index) : 인덱스에 있는 요소 제거
- size() : 리스트의 요소 갯수 반환
- contains(Object o) : 리스트에 특정 요소가 포함되어 있는지 확인
- clear() : 리스트의 모든 요소 제거

#### List 대표적인 클래스

- ArrayList : 배열 기반의 리스트로, 인덱스로 빠른 접근이 가능하지만 중간 삽입/삭제가 느림
- LinkedList : 노드 기반의 리스트로, 삽입/삭제가 빠르지만 인덱스를 통한 접근이 느림(메모리 연속적 X)
- Stack : 후입선출(LIFO) 구조를 가지며, 요소를 스택에 추가하고 제거하는 데에 사용
