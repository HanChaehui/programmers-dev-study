# 품질 높이기 (AOP 로깅 + 단위 테스트)

> 시리즈의 **마지막 편**이에요. 지금까지 만든 게시판이 "잘 도는지"를 눈과 코드로 **확인하는 힘**을 길러요. 모든 요청을 **AOP로 자동 로깅**하고, 핵심 로직을 **단위 테스트**로 자동 검증해요.
> **아래 Step을 순서대로 따라가면, 콘솔에 요청 로그가 찍히고 `./gradlew test`가 초록불로 통과하는 상태가 완성됩니다.**
>
> 각 Step의 **힌트는 접혀 있어요.** "이건 어느 계층을 어떻게 검증하지?"를 먼저 떠올리고, 막히면 "힌트 보기"를 펼치세요.
>
> 이번 편은 **1~5편이 끝나 게시판 CRUD와 파일 기능이 완성됐다는 전제**예요. 새 기능을 더하기보다, 이미 만든 코드에 **로깅과 테스트**를 입혀 품질을 올리는 게 목표예요.

---

## 0. 먼저 알아둘 점

- 이 과제는 **1~5편 완성본**(회원가입/로그인/게시판 목록·상세·쓰기·수정·삭제·파일)을 대상으로 해요. 그 코드를 검증 대상으로 삼아요.
- **AOP 의존성 주의(스프링 부트 4.1.0):** `spring-boot-starter-aop`가 잘 안 잡혀요. 그런데 우리는 이미 `spring-boot-starter-data-jpa`를 쓰고, 그 안의 `spring-aspects`를 통해 **`aspectjweaver`가 전이(transitive)로 딸려와요.** 그래서 **`build.gradle`에 AOP 의존성을 따로 추가하지 않아도** `@Aspect`가 동작해요.
- **테스트 도구는 스타터에 이미 포함**돼요: `spring-boot-starter-...-test`가 **JUnit 5 · Mockito · AssertJ · MockMvc**를 전부 끌어와요. 단, DB 슬라이스 테스트용 **H2는 Step 3에서 추가**해요.
- **여러분이 작성할 것:** 로깅 어스펙트 1개(`LoggingAspect`)와 계층별 테스트 3종(서비스/리포지토리/컨트롤러).

> 테스트의 목표는 "화면을 매번 손으로 눌러보지 않아도, **코드가 로직이 맞는지 스스로 증명**하게" 만드는 거예요. 한 번 짜두면 이후 리팩터링할 때 **안전망**이 돼요.

## 1. 무엇을 만드나요?

(1) 모든 컨트롤러 요청이 콘솔에 **자동 로깅**되고, (2) 핵심 로직이 **자동 테스트**로 검증돼요.

| 만드는 것 | 하는 일 | 계층/도구 |
|------|------|------|
| `LoggingAspect` | 모든 컨트롤러 호출 전/후를 로깅 | AOP (`@Aspect`) |
| `MemberServiceTest` | 서비스 로직 검증 (가짜 DB) | Mockito |
| `MemberRepositoryTest` | 쿼리 메서드 검증 (진짜 DB) | `@DataJpaTest` |
| `MemberApiControllerTest` | HTTP 계약 검증 | `@WebMvcTest` + MockMvc |

**완성되면 이렇게 동작해요**
```
[요청 로깅]
  브라우저에서 목록 요청 → 콘솔에:
  ===> 시작: BoardApiController.getBoardList(..)
  <=== 종료: BoardApiController.getBoardList(..) (12ms)

[테스트 실행]  ./gradlew test
  MemberServiceTest       ✓ 통과
  MemberRepositoryTest    ✓ 통과
  MemberApiControllerTest ✓ 통과
  BUILD SUCCESSFUL
```

## 3. 핵심 개념

### (1) AOP와 `@Aspect` — 공통 관심사 분리

로깅은 "모든 컨트롤러에 똑같이 끼워 넣고 싶은" 공통 작업이에요. 메서드마다 `System.out`을 넣으면 코드가 지저분해져요. **AOP**는 이런 공통 작업(관심사)을 **한 곳(Aspect)에 모아**, 대상 메서드 실행에 **자동으로 끼워 넣어**요.

| 용어 | 뜻 |
|------|------|
| `@Aspect` | 공통 작업을 모아둔 클래스 (로깅/트랜잭션/측정 등) |
| `@Pointcut` | "어디에 끼울지" 대상 지정 (예: controller 패키지 전체) |
| `@Around` | 대상 메서드 실행을 **앞뒤로 감싸는** 방식 (전/후 둘 다 가능) |
| `ProceedingJoinPoint` | 감싼 "그 대상 메서드" 자체. `proceed()`로 실제 실행 |

> 어떻게 자동으로 끼나요? 스프링이 대상 빈을 **프록시(대역)**로 감싸요. 우리가 컨트롤러를 부르면 사실은 프록시가 먼저 받아 **로깅 → 진짜 실행(`proceed`) → 로깅** 순서로 흘려줘요.

### (2) AOP vs `@RestControllerAdvice` — 언제 뭘?
| 상황 | 도구 |
|------|------|
| 로깅·실행시간 측정처럼 **아무데나 끼우는 범용 작업** | **AOP** (`@Aspect`) |
| 예외를 알맞은 HTTP 응답(409/404…)으로 바꾸는 **웹 특화 작업** | `@RestControllerAdvice` (1편에서 사용) |

### (3) 테스트 피라미드 — 계층마다 다른 방식으로
| 계층 | 도구 | 무엇을 보나 | 스프링 뜨나? | 속도 |
|------|------|------|------|------|
| ① 순수 단위 | Mockito | 서비스 로직 | ❌ 안 뜸 | 매우 빠름 |
| ② 리포지토리 슬라이스 | `@DataJpaTest` | 쿼리 메서드 | 일부(JPA만) | 보통 |
| ③ 웹 슬라이스 | `@WebMvcTest` | 컨트롤러/예외 | 일부(웹만) | 보통 |
| ④ 통합 | `@SpringBootTest` | 전체 흐름 | 전부 | 느림 |

> 위로 갈수록 빠르고 많이, 아래로 갈수록 느리고 적게. 이번엔 ①②③을 하나씩 만들어요.

### (4) Mockito — 가짜 객체(Mock)
- **Mock** = 진짜처럼 생겼지만 속은 빈 대역 배우. 내가 행동을 정해줄 수 있어요.
- 왜? 서비스만 콕 집어 보고 싶은데, 진짜 리포지토리를 쓰면 DB가 필요하고 느려요. 그래서 **가짜로 바꿔** 서비스 로직만 검증해요.
- **딱 2가지**: (1) 스터빙 `given(...).willReturn(...)` — "이렇게 부르면 이 값 줘라", (2) 검증 `verify(...)` — "그 메서드가 호출됐나".

### (5) 슬라이스 테스트 — "필요한 부품만" 뜬다
| | `@DataJpaTest` | `@WebMvcTest` |
|---|---|---|
| 뜨는 것 | JPA 리포지토리 + DB | 컨트롤러 + `@RestControllerAdvice` |
| 안 뜨는 것 | 컨트롤러/서비스 | 서비스/리포지토리 (→ `@MockitoBean`으로 가짜) |
| 특징 | 트랜잭션 자동 롤백 | 실제 서버 없이 MockMvc로 요청 흉내 |

## Step 1. AOP 개념 잡기 — 무엇을 감쌀지 정하기

로깅을 넣을 대상은 **모든 컨트롤러**예요. 메서드마다 손으로 넣지 않고, `controller` 패키지 전체를 한 번에 겨냥해요.

- `aop/LoggingAspect` 클래스를 만들고 `@Aspect @Component`를 붙여요.
- `@Pointcut`으로 "어디에 끼울지"를 `controller` 패키지로 지정해요.
- `build.gradle`에 AOP 의존성을 **추가하지 마세요.** (0번 참고 — `data-jpa`를 통해 이미 들어와 있어요.)

<details>
<summary>힌트 보기</summary>

```java
// aop/LoggingAspect.java
@Aspect        // "이 클래스는 공통 관심사(로깅)를 모아둔 곳"
@Component     // 스프링 빈으로 등록돼야 AOP가 동작한다
public class LoggingAspect {

    // controller 패키지 아래 "모든 클래스의 모든 메서드"를 대상으로 삼는다
    @Pointcut("execution(* com.example.spring.basicboard.controller..*(..))")
    public void controllerLayer() {}
}
```
> `execution(* ...controller..*(..))` = "controller 패키지(하위 포함)의 반환형 무관·인자 무관 모든 메서드".
> `@Pointcut`은 빈 메서드예요. 이름(`controllerLayer`)만 빌려 다음 Step에서 재사용해요.
</details>

---

## Step 2. `LoggingAspect` — `@Around`로 실행 전/후 로깅

이제 실제 로깅이에요. `@Around`는 대상 메서드를 **앞뒤로 감싸서**, 실행 전에 "시작", 실행 후에 "종료(+걸린 시간)"를 찍을 수 있어요.

- `@Around("controllerLayer()")` 메서드를 만들고 `ProceedingJoinPoint`를 받아요.
- `proceed()` **호출 전**에 시작 로그, **후**에 종료 로그를 `System.out`으로 찍어요. (학습용이라 로거 대신 sysout)
- `proceed()`의 **반환값을 그대로 return** 해야 원래 요청이 정상 동작해요.

<details>
<summary>힌트 보기</summary>

```java
// aop/LoggingAspect.java (이어서)
@Around("controllerLayer()")
public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
    String name = joinPoint.getSignature().toShortString(); // 예: BoardApiController.getBoardList(..)
    long start = System.currentTimeMillis();

    System.out.println("===> 시작: " + name);
    try {
        Object result = joinPoint.proceed();  // ★ 진짜 컨트롤러 메서드 실행
        return result;                         // 반환값을 그대로 돌려줘야 정상 동작
    } finally {
        long took = System.currentTimeMillis() - start;
        System.out.println("<=== 종료: " + name + " (" + took + "ms)");
    }
}
```
> `proceed()`를 빼먹으면? 대상 메서드가 실행되지 않아 화면이 텅 비어요. AOP는 "감싸는" 것이지 "대신하는" 게 아니에요.
> `finally`에 종료 로그를 두면, 예외가 나도 종료 로그는 찍혀요.
</details>

## Step 3. 테스트 세팅 — 인메모리 H2 준비하기

테스트는 매번 **깨끗하고 빠른 DB**가 필요해요. 운영 MySQL 대신 **인메모리 H2**를 테스트 전용으로 붙여요.

- `build.gradle`에 H2를 **테스트 런타임**으로 추가해요.
- `src/test/resources/application.yaml`을 만들어 H2(MySQL 호환 모드) + `ddl-auto: create-drop` + `sql.init: never` + `file.upload-dir`을 설정해요.

<details>
<summary>힌트 보기</summary>

```gradle
// build.gradle - dependencies 안에 추가
testRuntimeOnly 'com.h2database:h2' // @DataJpaTest 용 인메모리 DB (테스트 전용)
```
```yaml
# src/test/resources/application.yaml  (테스트 실행 때만 적용)
file:
  upload-dir: ./build/test-uploads   # FileService의 @Value 주입값 (없으면 전체 컨텍스트 로딩 실패)
spring:
  datasource:
    url: jdbc:h2:mem:testdb;MODE=MySQL;DB_CLOSE_DELAY=-1  # MySQL 문법 호환 인메모리 H2
    driver-class-name: org.h2.Driver
    username: sa
    password:
  jpa:
    hibernate:
      ddl-auto: create-drop          # 엔티티로 테이블 생성 후 종료 시 삭제
  sql:
    init:
      mode: never                    # data.sql(운영 시드)은 테스트에서 실행 안 함
```
> H2의 두 모드: 지금은 **임베디드 모드**(`jdbc:h2:mem:` — 테스트 JVM 안에서 잠깐 뜸)예요. JPA 수업에서 쓰던 `jdbc:h2:tcp://...`는 **서버(TCP) 모드**(독립 프로세스라 웹 콘솔로 따로 접속 가능)예요. 테스트엔 매번 깨끗한 임베디드가 알맞아요.
</details>

## Step 4. 계층 ① — 서비스 단위 테스트 (Mockito)

`MemberService`만 진짜로 쓰고, 그것이 의존하는 리포지토리·매퍼는 **가짜(Mock)**로 바꿔 로직만 검증해요.

- `@ExtendWith(MockitoExtension.class)` + `@Mock`(리포지토리/매퍼) + `@InjectMocks`(서비스).
- `login`: 성공 / 비밀번호 틀림(빈 Optional) / 없는 아이디(빈 Optional).
- `join`: 성공(`verify(save)`) / 중복(`assertThatThrownBy`로 예외 확인, `verify(never())`로 저장 안 됨).

> 📎 **Mockito 치트시트:** `given(mock.메서드(인자)).willReturn(값)`(스터빙) · `verify(mock).메서드()`(호출됨) · `verify(mock, never()).메서드()`(호출 안 됨) · `any()`(아무 인자나).

<details>
<summary>힌트 보기</summary>

```java
// src/test/java/.../service/MemberServiceTest.java
@ExtendWith(MockitoExtension.class)
class MemberServiceTest {

    @Mock MemberRepository memberRepository; // 가짜 (DB 안 씀)
    @Mock MemberMapper memberMapper;
    @InjectMocks MemberService memberService; // 대상 (위 Mock들이 주입됨)

    @Test
    void login_성공() {
        Member member = Member.builder().userId("hong").password("1234").userName("홍길동").build();
        // (1) member 객체를 만든 것과 (2) 가짜가 그걸 돌려주게 하는 건 별개다!
        given(memberRepository.findByUserId("hong")).willReturn(Optional.of(member));

        LoginRequestDto req = new LoginRequestDto();
        req.setUsername("hong"); req.setPassword("1234");

        Optional<Member> result = memberService.login(req);
        assertThat(result).isPresent();
    }

    @Test
    void join_중복이면_예외() {
        MemberJoinRequestDto req = new MemberJoinRequestDto();
        req.setUserId("hong");
        given(memberRepository.existsByUserId("hong")).willReturn(true);

        assertThatThrownBy(() -> memberService.join(req))
                .isInstanceOf(DuplicateUserIdException.class);
        verify(memberRepository, never()).save(any()); // 중복이면 저장 안 함
    }
}
```
> 스프링을 안 띄우므로(순수 `new` 객체) 클래스 레벨 `@Transactional`은 **여기선 동작하지 않아요.** 트랜잭션은 스프링 프록시 기능이거든요.
> `login`이 내부에서 `findByUserId`를 부르는데, 그 호출이 가짜에게 가요. 그래서 `given`으로 미리 답을 심어야 해요.
</details>

## Step 5. 계층 ② — 리포지토리 테스트 (`@DataJpaTest`)

쿼리 메서드(`existsByUserId`, `findByUserId`)는 "이름 → SQL" 자동 변환이 핵심이라, **Mock이 아니라 진짜 DB**로 검증해야 해요.

- `@DataJpaTest` + `@AutoConfigureTestDatabase(replace = NONE)` (Step 3에서 만든 H2를 그대로 사용).
- `@BeforeEach`에서 회원 1명 저장 → `existsByUserId` true/false, `findByUserId` 있음/없음 검증.
- **스프링 부트 4는 import 경로가 3.x와 달라요.** (아래 힌트의 패키지 그대로 써요.)

<details>
<summary>힌트 보기</summary>

```java
// src/test/java/.../domain/repository/MemberRepositoryTest.java
import org.springframework.boot.data.jpa.test.autoconfigure.DataJpaTest;             // 부트4 경로
import org.springframework.boot.jdbc.test.autoconfigure.AutoConfigureTestDatabase;    // 부트4 경로

@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE) // 내가 지정한 H2 사용
class MemberRepositoryTest {

    @Autowired MemberRepository memberRepository;

    @BeforeEach
    void setUp() {
        memberRepository.save(Member.builder()
                .userId("hong").password("1234").userName("홍길동").build());
    }

    @Test
    void existsByUserId_존재하면_true() {
        assertThat(memberRepository.existsByUserId("hong")).isTrue();
        assertThat(memberRepository.existsByUserId("nobody")).isFalse();
    }

    @Test
    void findByUserId_있으면_회원() {
        assertThat(memberRepository.findByUserId("hong")).isPresent();
        assertThat(memberRepository.findByUserId("nobody")).isEmpty();
    }
}
```
> `@DataJpaTest`는 JPA 관련 부품만 뜨고(가벼움), 각 테스트는 **끝나면 자동 롤백**돼서 서로 데이터가 안 섞여요.
</details>

## Step 6. 계층 ③ — 컨트롤러 테스트 (`@WebMvcTest`) + 최종 확인

컨트롤러의 **HTTP 계약**(상태코드·JSON·예외 변환)을 MockMvc로 검증해요. 서비스는 `@MockitoBean`으로 가짜를 넣어요.

- `@WebMvcTest(MemberApiController.class)` + `@MockitoBean MemberService` + `MockMvc`.
- 성공: `POST /api/members/join` → 200 + `$.url == "/members/login"`.
- 중복: 서비스가 `willThrow(...)` → `GlobalExceptionHandler`가 **409 + message**로 변환하는지.
- 마지막에 `./gradlew test`로 **전부 초록**인지 확인!

<details>
<summary>힌트 보기</summary>

```java
// src/test/java/.../controller/MemberApiControllerTest.java
import org.springframework.boot.webmvc.test.autoconfigure.WebMvcTest;               // 부트4 경로
import org.springframework.test.context.bean.override.mockito.MockitoBean;           // 구 @MockBean 대체

@WebMvcTest(MemberApiController.class)
class MemberApiControllerTest {

    @Autowired MockMvc mockMvc;
    @MockitoBean MemberService memberService; // 컨트롤러가 의존하는 서비스를 가짜로

    @Test
    void join_성공() throws Exception {
        String json = """
                {"userId":"newbie","password":"1234","userName":"새싹"}
                """;
        mockMvc.perform(post("/api/members/join")
                        .contentType(MediaType.APPLICATION_JSON).content(json))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.url").value("/members/login"));
    }

    @Test
    void join_중복이면_409() throws Exception {
        willThrow(new DuplicateUserIdException("이미 존재하는 아이디입니다."))
                .given(memberService).join(any());
        String json = """
                {"userId":"hong","password":"1234","userName":"홍길동"}
                """;
        mockMvc.perform(post("/api/members/join")
                        .contentType(MediaType.APPLICATION_JSON).content(json))
                .andExpect(status().isConflict())               // 409
                .andExpect(jsonPath("$.message").value("이미 존재하는 아이디입니다."));
    }
}
```
> `@WebMvcTest`는 `@RestControllerAdvice`(GlobalExceptionHandler)를 **자동으로 함께 로드**해서, 예외 → 409 변환까지 검증돼요.
> JSON은 `ObjectMapper` 없이 **문자열 리터럴**로 보내면 요청 본문이 눈에 보여 더 명확해요.
> void 메서드가 예외를 던지게 할 땐 `willThrow(...).given(mock).메서드()` 순서예요.
</details>
